# AnimeTranscoder CLI 与本地语音筛选功能评估

## 1. 文档目的

这份文档用于评估两个彼此相关、但不能混为一谈的方向：

- 方向一：为 `AnimeTranscoder` 建立正式的 `CLI` 控制能力，让命令行代理或脚本能够稳定驱动软件
- 方向二：在此基础上，增加“从视频中尽量保留目标对白、去除不需要声音”的工作流

这份文档的重点不是直接进入某个模型或某个工具选型，而是先把产品边界、工程顺序和技术难点说清楚。

## 2. 先说结论

你的整体判断大方向是对的：

- 应该先做 `CLI` 控制层
- 后做语音文本识别与筛选工作流
- 不应该一开始就在软件内部塞进大量云端 AI 逻辑

但这里有一个非常关键的修正：

真正应该优先做的，不是“让 `Codex` 或 `Claude Code` 像人一样去盲点 GUI”，而是“让软件本身暴露稳定、结构化、可脚本调用的命令入口”。

也就是说：

- 正确方向是 `CLI / headless workflow`
- 不是把“屏幕自动化”当成主控制方案

屏幕自动化可以作为测试手段或兜底手段，但不能作为主产品架构。

## 3. 为什么不能把“AI 像人点界面”当主方案

表面上看，让代理像人一样点软件似乎很灵活，但从工程上看它有明显问题：

- 对界面布局非常脆弱
- 一改按钮文案或控件位置就容易失效
- 很难做稳定日志和错误码
- 很难做批处理
- 很难做参数版本管理
- 很难让不同代理稳定复现同一结果

而 `CLI` 方案的优点正好相反：

- 参数固定
- 输入输出清晰
- 容易记录日志
- 容易做失败分类
- 容易被脚本、代理、调度器复用
- 更适合批处理和长期维护

因此，后续如果要让命令行代理来“使用这个软件”，最好的方式不是去看屏幕点按钮，而是调用：

- `AnimeTranscoder.exe probe ...`
- `AnimeTranscoder.exe audio extract ...`
- `AnimeTranscoder.exe transcript export ...`
- `AnimeTranscoder.exe audio render-selection ...`

代理只负责决策和编排，软件自己负责确定性执行。

## 4. 对当前项目现状的判断

结合当前仓库，可以得出几个很重要的现实结论。

### 4.1 当前项目已经有较多可复用的服务层

当前项目不是单纯的按钮拼接器，而是已经把不少核心能力放进了 `Services/`：

- 媒体探测
- 音轨解析
- 音频提取
- 静音检测
- 视频裁剪
- 任务执行
- 输出校验

这意味着未来增加 `CLI` 时，不需要推翻 GUI 重写一套媒体逻辑。

更合理的做法是：

- 保留现有服务层
- 把 `MainViewModel` 里一部分工作流编排抽出去
- 再在 GUI 和 CLI 两边共同调用

### 4.2 当前项目还没有正式 CLI 入口

当前程序仍然是一个标准 `WPF WinExe` 应用，`App.xaml.cs` 里只是普通启动，没有看到正式命令行参数入口。

这说明：

- 现在还不能稳定通过命令行控制它
- 也没有机器可读的输出协议
- 代理目前若要“操作软件”，只能依赖 GUI 自动化或直接改代码

### 4.3 当前项目已经有音频处理基础，但还没有“语义筛选”基础

项目现在已经有：

- 音轨探测
- 音频抽取
- 静音检测
- 混音

但还没有下面这些与目标功能直接相关的能力：

- 语音转文字结果管理
- 时间轴文本交换格式
- 片段级保留/剔除规则
- 说话人分离结果管理
- 声源分离结果管理

所以如果直接跳到“保留游戏对白、去掉主播声音”，中间会缺一整层能力。

## 5. 你的需求真正难在哪里

你现在描述的目标，本质上不是一个简单的“音轨提取”功能，而是一个“语义驱动的音频筛选工作流”。

这里要分清三种完全不同的问题。

### 5.1 问题一：多音轨选择

如果视频本身就已经有多条独立音轨，例如：

- 音轨 1：主播麦克风
- 音轨 2：游戏原声

那这件事很简单。

软件只需要：

- 探测音轨
- 让用户或代理选择目标音轨
- 直接导出目标音轨

这类场景主要是工程问题，不是 AI 问题。

### 5.2 问题二：单音轨里有多个不同说话人

如果主播声音和游戏角色对白都混在同一条音轨里，但它们通常轮流说话，不经常重叠，那么事情会难一些。

这时仅靠“转文字”是不够的。

因为语音识别只会告诉你：

- 某个时间说了什么

但它未必能稳定告诉你：

- 这句话是谁说的
- 这是主播在说，还是游戏角色在说

这时你需要的是：

- 语音识别
- 说话人分离或说话人标注
- 再结合文本内容做筛选

### 5.3 问题三：多个声音重叠在一起

如果主播在说话的同时，游戏角色也在说话，甚至还叠加背景音乐和音效，那么这是最难的一类。

这时单纯依靠时间轴文本筛选是做不到“干净剔除”的。

因为同一时间段里，不同声音已经混在一条波形里了。

此时需要的是：

- 声源分离
- 人声/伴奏分离
- 可能还需要说话人分离

而且即便这样，效果也未必完全干净。

所以必须提前接受一个现实：

“只保留游戏人物剧情对话，完全去掉主播声音”这件事，在很多直播素材里并不是一个百分之百可逆、可精确实现的问题。

它更像是：

- 某些素材效果很好
- 某些素材效果一般
- 某些素材本身就做不到理想结果

## 6. 因此，这个功能不能只靠“音轨切出来 + 转文字”

这是这次评估里最重要的技术结论之一。

如果未来真要实现你说的工作流，至少要把处理链拆成下面几层：

1. 媒体层：把目标音轨或混合音频提取出来
2. 识别层：把语音转成带时间轴的文本
3. 分离层：尽可能识别说话人或声源
4. 决策层：由人或外部代理决定哪些片段保留
5. 渲染层：按规则重新输出目标音频

如果只做到第 1 层和第 2 层，那么你能得到的是：

- 一个 transcript
- 一份时间轴

但还不够支撑“稳定保留游戏对白、剔除主播声音”。

## 7. 对“AI 在系统中的位置”的建议

你的边界判断是合理的：

- 不要把云端 AI 深度耦合进这个软件
- 软件本身尽量保持本地、确定性、可维护

我建议把 AI 的位置定义成：

- 软件内部不负责“大模型推理”
- 软件内部只负责工作流编排、文件交换、参数执行
- 文本理解、语义判断、对白筛选可以由外部代理负责
- 本地语音转文字可以由本地可执行工具或本地模型负责

换句话说，这个软件未来更适合做：

- workflow host
- media executor
- transcript exchange hub

而不是做：

- 一体化 AI 平台

## 8. 推荐的总体架构

我建议把后续架构分成四层。

### 8.1 表现层

包括：

- `WPF GUI`
- 后续的 `CLI`

这两个入口都不应该直接承担复杂媒体逻辑。

### 8.2 工作流编排层

新增一层真正的工作流服务，例如：

- `MediaProbeWorkflow`
- `AudioWorkflow`
- `TranscriptWorkflow`
- `SelectionRenderWorkflow`

这层负责：

- 参数校验
- 默认规则
- 阶段控制
- 结果归档
- 错误分类

### 8.3 执行层

继续复用当前已有的：

- `ffprobe`
- `ffmpeg`
- 原生 `AnimeMediaCore`
- 各种本地可执行工具

### 8.4 外部智能层

这层不必内嵌到软件里。

它可以是：

- 本地语音识别工具
- 本地说话人分离工具
- 外部命令行代理

它们通过文件协议与软件交互，而不是直接嵌进主程序内部。

### 8.5 项目结构上的推荐落地方式

这里还需要把“架构分层”和“项目拆分”分开看。

结合当前仓库现状，更推荐的不是在现有 `WPF WinExe` 里硬塞一套控制台模式，而是逐步拆成下面这种结构：

```text
AnimeTranscoder/
  AnimeTranscoder.App/        # WPF GUI
  AnimeTranscoder.Cli/        # Console / CLI
  AnimeTranscoder.Core/       # Workflow + Services + Models
  AnimeTranscoder.Tests/
```

职责建议如下：

- `AnimeTranscoder.App` 只负责 GUI、绑定、对话框、界面生命周期
- `AnimeTranscoder.Cli` 只负责参数解析、stdout/stderr、退出码、Ctrl+C
- `AnimeTranscoder.Core` 负责可复用工作流、服务编排、数据模型、错误定义

这样做的原因很直接：

- 当前项目仍是 `WPF` 主程序
- `WinExe` 并不是最自然的 CLI 宿主
- 如果后续要稳定输出 `json`、进度流、取消信号，单独的 CLI 项目通常更干净
- GUI 和 CLI 共用同一套 Core，能避免复制工作流代码

因此，推荐的技术路径不是“先想办法让 WPF 冒充控制台程序”，而是：

1. 先抽 `Core`
2. 再加独立 `Cli`
3. 最后让 `App` 和 `Cli` 同时依赖 `Core`

## 9. CLI 落地时必须提前定下来的实现约束

报告如果只停留在“做 CLI”这一层，后续很容易在实现上反复返工，所以有几个约束最好现在就写死。

### 9.1 长任务的输出协议

转码、音频提取、声学分析都属于长任务，CLI 至少要支持三类输出：

- 最终结果 `json`
- 过程日志
- 进度事件

比较实用的规则是：

- `stdout` 默认只输出机器可读结果
- `stderr` 输出人类可读日志
- 需要流式进度时，用 `--progress jsonl` 或类似参数持续输出事件

事件可以最小化定义为：

- `stage`
- `progress`
- `speed`
- `message`
- `timestamp`

### 9.2 取消与退出码

CLI 不是只要“能跑完”就够了，还要能安全中断。

建议最少定义这些行为：

- `Ctrl+C` 触发取消令牌
- 优雅终止 `ffmpeg` 子进程
- 删除或标记未完成的临时文件
- 区分“失败”和“用户取消”两种退出码

### 9.3 中间音频格式

“导出音频为标准中间文件”不能只是一句口号，必须明确约定。

如果目标主要是本地语音识别，第一版更建议统一成：

- `wav`
- `mono`
- `16kHz` 或 `24kHz`
- PCM

这样做的优点是：

- 几乎所有本地转写工具都能稳定吃进去
- 不引入有损压缩误差
- 后续脚本处理最简单

缺点是文件较大，所以还需要同时规定：

- 临时目录位置
- 是否允许保留中间件
- 成功后是否自动清理

如果未来要兼顾体积，可以再考虑增加 `flac` 作为可选中间格式。

## 10. 为什么“先做 CLI”这个顺序是对的

这个顺序是对的，原因不只是“方便 AI 用”，更重要的是它会直接改善项目结构。

### 10.1 CLI 会倒逼你把 GUI 里的流程抽干净

现在不少工作流编排还在 `MainViewModel` 里。

一旦要做 `CLI`，就必须明确：

- 哪些是界面状态
- 哪些是业务参数
- 哪些是可复用工作流

这会促使项目从“GUI 驱动业务”走向“业务驱动 GUI/CLI”。

### 10.2 CLI 会自然形成机器可读协议

如果将来要让代理自动执行，就必须有稳定协议，例如：

- 退出码
- `json` 输出
- 阶段化日志
- 明确错误信息

这比做一个“能点按钮”的界面自动化可靠得多。

### 10.3 CLI 会直接提升批处理价值

即便不接 AI，`CLI` 本身也有价值：

- 可以写批处理脚本
- 可以做计划任务
- 可以被其他本地工具调用
- 可以做流水线式组合

也就是说，`CLI` 不是为了 AI 才存在，而是产品工程本身就该有的一层。

## 11. 推荐的分阶段落地方案

下面给出一个更现实的开发顺序。

## 11.1 第一阶段：建立正式 CLI 基础层

这一阶段的目标不是做语音筛选，而是先把软件从“只能手工点界面”升级成“可被命令行稳定调用”。

建议先支持少量高价值命令：

- `probe`
- `audio extract`
- `audio detect-silence`
- `queue transcode`

建议规则：

- 输入参数清晰
- 标准输出支持 `json`
- 标准错误输出用于人类可读日志
- 退出码稳定
- 不弹 GUI 对话框
- 不依赖界面线程交互

这一阶段还有一个必须落地的结构性动作：

- 把现有 `MainViewModel` 中的业务编排拆进 `Core` 工作流

否则后面的 CLI 很容易变成：

- GUI 一套流程
- CLI 一套流程

这会直接把维护成本翻倍。

第一阶段完成后，就已经可以让外部代理开始“通过命令行使用软件”。

## 11.2 第二阶段：建立 transcript 交换能力

这一阶段先不急着做筛选，而是先建立“文本和时间轴”的正式数据模型。

建议软件支持：

- 导出音频为标准中间文件
- 导入本地转写结果
- 保存 transcript `json`
- 导出 `srt/vtt/txt`

最关键的是先把 transcript 数据结构固定下来，例如每条 segment 至少包含：

- `id`
- `start`
- `end`
- `text`
- `language`
- `speaker`
- `confidence`
- `source`
- `overlap`
- `channel`

其中：

- `speaker` 可以先允许为空
- `overlap` 用于标记多人重叠说话或高风险片段
- `channel` 允许未来接多声道或分轨结果

推荐最小草案如下：

```json
{
  "version": "1.0",
  "mediaPath": "input.mp4",
  "audioPath": "work/audio/input.wav",
  "segments": [
    {
      "id": "seg-0001",
      "start": 12.34,
      "end": 14.82,
      "text": "这里是剧情对白",
      "language": "ja",
      "speaker": null,
      "confidence": 0.91,
      "source": "whisper.cpp",
      "overlap": false,
      "channel": "mono"
    }
  ]
}
```

### 11.2.1 本地工具生态要尽早纳入设计

这里不能只说“接本地工具”，最好从一开始就按主流工具来设计兼容层。

当前值得优先参考的本地方案包括：

- 语音识别：`whisper.cpp`
- 语音识别：`faster-whisper`
- 声源分离：`Demucs`
- 说话人分离：`pyannote.audio`

这样做的意义是：

- 可以直接参考它们已有的 CLI 行为和输出格式
- transcript 导入层不会变成凭空设计
- 后面真接入时更容易落地

## 11.3 第三阶段：建立“选择结果”协议

这是你设想里最关键的一步。

也就是：

- 软件负责导出 transcript
- 外部代理读取 transcript
- 外部代理产出“保留哪些片段”的选择结果
- 软件再根据选择结果渲染输出

这里建议定义一个选择文件，例如 `selection.json`，内容包括：

- `version`
- `transcriptVersion`
- `targetSegments`
- 保留区间
- 剔除区间
- 原因
- 决策来源
- 冲突处理策略
- 不确定片段

这样做的好处是：

- 软件不负责理解语义
- 代理不负责实际媒体渲染
- 两边职责清晰

但这里不能只停在概念层，还要把规则写清楚。

至少建议明确：

- `segment id` 优先于纯时间区间引用
- 如果“保留”和“剔除”冲突，默认以更保守的规则为准
- 允许 `uncertain` 片段进入 GUI 二次确认
- 协议必须带版本号

推荐最小草案如下：

```json
{
  "version": "1.0",
  "transcriptVersion": "1.0",
  "decisionSource": "external-agent",
  "conflictPolicy": "exclude_wins",
  "targetSegments": [
    {
      "segmentId": "seg-0001",
      "action": "keep",
      "reason": "剧情主线对白"
    },
    {
      "segmentId": "seg-0002",
      "action": "uncertain",
      "reason": "可能与主播串音重叠"
    }
  ]
}
```

## 11.4 第四阶段：建立音频片段渲染能力

当有了选择结果，软件再负责真正输出目标音频。

这一阶段建议先做最稳、最容易落地的版本：

- 按时间段保留
- 非保留区间静音
- 非保留区间直接剪掉
- 相邻片段自动淡入淡出

注意这里有两个不同输出模式，必须分开：

- 模式 A：保留原时间轴，只把不需要部分静音
- 模式 B：重新拼接，只输出保留对白的连续成片

这两个模式价值都很高，但适用场景不同。

这里还要补一个现实约束：

- 裁剪、拼接、淡入淡出、重采样、编码格式转换都要统一由渲染层处理

不要让外部代理去控制这些底层媒体细节，否则职责会重新混乱。

## 11.5 第五阶段：视效果再决定是否做说话人/声源分离集成

只有当前四阶段都跑通以后，才值得继续往下做：

- 说话人分离结果导入
- 声源分离结果导入
- 主播声/角色声候选打分

原因很简单：

- 这部分技术难度高
- 样本差异大
- 成本高
- 成功率不稳定

不能把最难的一层放在第一步。

## 12. Service 与 Workflow 的边界建议

为了避免后面做出两层含义重复的抽象，必须先把这两个词的边界说清楚。

推荐定义如下：

- `Service`：单一职责能力单元
- `Workflow`：多个 `Service` 的组合编排

例如：

- `FfprobeService` 负责探测
- `AudioExtractionService` 负责提取
- `TranscriptWorkflow` 负责“提取中间音频 -> 导入 transcript -> 保存项目状态”
- `SelectionRenderWorkflow` 负责“读取 selection -> 生成渲染计划 -> 输出结果”

这样做的好处是：

- 不会把现有 `Services/` 全部推翻
- 可以在现有工程基础上增量演进
- CLI 和 GUI 都更容易复用相同工作流

## 13. 功能边界建议

为了避免这个项目后面越做越散，建议提前规定边界。

### 13.1 软件内部建议负责的内容

- 媒体探测
- 音轨提取
- 中间文件管理
- transcript 导入导出
- 选择结果导入导出
- 音频渲染
- 日志
- 历史记录
- 批处理
- GUI 与 CLI 共用的工作流

### 13.2 软件内部暂时不要负责的内容

- 大模型提示词工程
- 云端模型调用
- 复杂语义理解
- 复杂说话人分类策略
- 代理框架本身

这些内容可以放在软件外部。

之所以现在不把它们塞进主程序，不只是“怕麻烦”，而是因为：

- 依赖变化太快
- 调试链路太长
- 会把本地媒体工具变成 AI 集成平台
- 一旦出问题，很难判断是模型问题还是媒体流程问题

因此第一版要有明确否决记录：

- 软件先做可确定执行
- AI 只做可替换的外部决策层

## 14. 一个更现实的最终工作流

如果按照上面的边界设计，未来一个比较清晰的完整链路会是这样：

1. 用户或代理调用 `CLI` 让软件探测视频
2. 软件导出目标音轨或混合音频
3. 本地转写工具把音频转成 transcript
4. 软件或外部脚本把 transcript 保存为标准 `json`
5. 外部代理读取 transcript，挑选与游戏剧情对白相关的文本段
6. 外部代理输出 `selection.json`
7. 软件读取 `selection.json`
8. 软件按时间段重新渲染音频
9. 软件输出结果并生成日志/历史记录

这个链路的优点是：

- 软件和 AI 的职责分离
- GUI 和 CLI 都能复用
- 后续可以逐步增强，但不会一次性耦死

这里建议再把“文件协议”说得更具体一些。

与其让外部工具随意在目录里读写，不如尽量走下面这种形式：

```powershell
AnimeTranscoder.Cli.exe project init --input input.mp4 --project demo.atproj
AnimeTranscoder.Cli.exe audio export-work --project demo.atproj
AnimeTranscoder.Cli.exe transcript import --project demo.atproj --input transcript.json
AnimeTranscoder.Cli.exe selection import --project demo.atproj --input selection.json
AnimeTranscoder.Cli.exe audio render-selection --project demo.atproj --output result.wav
```

也就是说：

- 命令本身尽量无状态
- 上下文放进项目文件
- transcript 和 selection 都作为显式导入物

这样会比“约定某个目录里有一些文件”更稳定。

## 15. 需要提前提醒的几个风险

### 15.1 文本正确，不等于音频可干净分离

就算 transcript 完全正确，只要声音重叠严重，也未必能得到理想音频结果。

缓解建议：

- 允许 `selection` 标记高风险片段
- GUI 提供二次确认
- 第一版优先支持“静音非目标区间”而不是追求彻底净化

### 15.2 “识别谁在说话”本身就是高难问题

有些游戏角色和主播可能同为人声，而且音色接近。

单靠文本内容判断容易误杀或漏保留。

缓解建议：

- 尽早引入 `speaker` 可为空的设计
- 允许接入外部分离结果，但不把它作为强依赖
- 让外部代理能输出 `uncertain`

### 15.3 不同素材差异会非常大

直播素材、录播素材、动画素材、游戏剧情录像，它们的音频结构完全不同。

必须允许这个功能是“效果型功能”，而不是“绝对正确功能”。

缓解建议：

- 先选一类样本做第一版，比如“单音轨、主播和游戏对白轮流出现的直播片段”
- 不要用最复杂素材定义第一版成败

### 15.4 如果一开始就把 AI 内嵌进主程序，维护会迅速失控

这会导致：

- 依赖复杂
- 调试复杂
- 版本管理复杂
- 本地环境要求复杂

因此第一版一定要坚持“软件负责流程，AI 负责判断”的边界。

缓解建议：

- AI 交互先做文件导入导出
- 不做主程序内的模型生命周期管理
- 不把提示词和模型配置绑死在仓库主逻辑里

## 16. Roadmap 说明

上面的“阶段”是时间顺序，不是另一套独立的优先级系统。

如果只看近期开发，可以把它压缩理解成：

- `P0`：抽 `Core`、落第一批 CLI、定输出协议
- `P1`：定 transcript/selection schema，接入导入导出
- `P2`：做渲染与项目文件串联
- `P3`：再评估分离工具集成

也就是说：

- 五阶段用于说明依赖顺序
- `P0-P3` 只是对应的近期实施优先级

## 17. 对当前项目的直接开发建议

如果下一步真的开始做，我建议优先顺序如下。

### P0

- 把 `MainViewModel` 中可复用流程抽到独立工作流服务
- 建立 `AnimeTranscoder.Core`
- 建立 `AnimeTranscoder.Cli`
- 设计 CLI 命令入口
- 设计统一 `json` 输出格式
- 设计错误码和日志约定

### P1

- 给音频提取能力补正式 CLI
- 给音轨探测能力补正式 CLI
- 定义 transcript 数据模型
- 定义 selection 数据模型

### P2

- 做 transcript 导入导出
- 做按时间区间静音/裁剪/拼接输出
- 做历史记录与结果文件归档

### P3

## 18. 最终判断

如果只用一句话概括这次评估：

你提出的方向是成立的，但真正的第一步不是“做 AI 音频筛选”，而是“先把软件变成一个可被命令行稳定驱动的工作流执行器”。

而对于“只保留游戏人物剧情对白”的目标，也必须明确：

- 单靠音轨提取不够
- 单靠转文字也不够
- 真正可用的方案应该是“本地转写 + 可选分离 + 外部代理选择 + 软件负责渲染”

只要把这个边界立住，后续开发就会清晰很多。

## 19. 结合当前仓库的补充说明

结合当前代码结构，这个方向在工程上是可行的，原因主要有三点：

- 当前项目已经有较完整的 `Services/` 层，不是所有逻辑都写死在界面里
- 当前已经具备音轨探测和音频抽取基础，说明“音频处理中间层”并非空白
- 当前还没有正式 CLI 入口，因此现在正适合把这层补上，而不是等功能越来越多后再返工

但也必须看到当前代码的现实：

- 项目仍是 `WPF` 主程序
- `MainViewModel` 仍然承担了大量编排职责
- 如果不先做工作流抽离，未来 CLI 很容易变成另一份重复代码

因此，第一步最应该做的事情不是研究某个 AI 模型，而是做结构整理。
