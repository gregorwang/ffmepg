# AnimeTranscoder 产品升级方案

## 1. 文档目标

这份文档用于把当前 `AnimeTranscoder` 从一个 `WPF + ffmpeg CLI` 的可用原型，升级成一套面向番剧处理工作流的桌面产品，并为后续的 `C/C++` 原生媒体核心预留正式演进路线。

目标不是做一个“万能格式工厂”，而是做一个在特定工作流上明显优于通用软件的工具：

- 把动漫视频拖进去
- 自动识别合适字幕轨
- 自动转成 `MP4 + H.264 + AAC LC`
- 自动优先使用 GPU
- 自动生成稳定成品

## 2. 产品定位

### 2.1 一句话定位

`AnimeTranscoder` 是一款面向番剧压制与整理场景的 Windows 原生桌面工具，核心价值是把固定、高频、重复的本地转码流程产品化。

### 2.2 用户价值

相比通用转码软件，本产品的价值不在“支持一切格式”，而在“把你的固定流程做到足够省心”。

核心价值包括：

- 中文文件名和中文路径稳定可用
- 自动识别内嵌字幕并选择目标字幕轨
- 自动烧录硬字幕
- 自动转为统一媒体格式
- 自动使用 `NVENC`，失败时自动回退 CPU
- 自动跳过已完成文件并做输出校验
- 尽量减少手工点选参数

### 2.3 不做什么

当前阶段明确不做：

- 专业剪辑
- 时间轴编辑
- 调色
- 播放器
- 全格式万能封装工厂
- 云端转码平台

## 3. 当前基础

当前项目已经具备这些基础能力：

- `WPF + .NET 8` 桌面程序骨架
- 中文化界面
- 文件拖拽与目录扫描
- 任务队列
- 设置持久化
- `ffprobe` 探测
- 字幕轨自动选择
- `ffmpeg` 命令拼装
- `NVENC` 检测与回退
- 输出时长校验

当前核心代码位于：

- `Views/`
- `ViewModels/`
- `Services/`
- `Models/`

这意味着产品第一阶段不需要推翻重做，而是继续在现有项目上演进。

## 4. 为什么这不是无意义的重复造轮子

市面上确实存在大量基于 `ffmpeg` 的图形化工具，但它们大多属于两类：

- 通用型转码器
- 专业后期工具中的转码功能

`AnimeTranscoder` 应该走第三类路线：

- 工作流产品

也就是围绕固定流程做深，而不是围绕“支持更多格式”做宽。

只要产品边界清晰，这个项目就不是在和所有专业软件正面竞争，而是在把你自己的高频流程沉淀成一款更适合你的工具。

## 5. 升级后的总体架构

推荐采用分层架构：

### 5.1 应用层

技术建议：

- `C#`
- `.NET 8`
- `WPF`

职责：

- 界面
- 设置管理
- 任务队列
- 结果展示
- 日志面板
- 错误提示
- 文件拖拽
- 目录扫描

### 5.2 工作流层

技术建议：

- `C#`

职责：

- 媒体探测调度
- 字幕策略决策
- 编码策略决策
- 重试策略
- 文件命名规则
- 成品校验
- 批量任务编排

这层是产品差异化的重点。

### 5.3 媒体执行层

第一阶段技术建议：

- `ffmpeg CLI`
- `ffprobe CLI`

第二阶段技术建议：

- 原生 `DLL`
- `libavformat`
- `libavcodec`
- `libavfilter`
- `libswscale`
- `libswresample`

职责：

- 探测媒体流
- 执行转码
- 生成滤镜图
- 执行字幕烧录
- 处理编码器与封装器参数

### 5.4 原生核心层

技术建议：

- `C` 导出接口
- `C++` 内部实现

职责：

- 对外提供稳定 ABI
- 对内封装复杂媒体逻辑
- 逐步替代部分 CLI 调用

推荐结构：

```text
AnimeTranscoder/
  docs/
  native/
    AnimeMediaCore/
      include/
      src/
      tests/
  Config/
  Infrastructure/
  Models/
  Services/
  ViewModels/
  Views/
```

## 6. 为什么要新增原生模块

如果未来要深入到底层控制开发，只靠 `C#` 拼命令是不够的。

新增 `native/AnimeMediaCore` 的目的不是马上推翻当前 CLI 方案，而是提前建立一条可扩展的技术边界：

- `C#` 继续做产品层
- `C/C++` 只做媒体核心
- 两者通过 `DLL` 接口通信

这样做有几个好处：

- 避免未来大重构
- 保持 WPF 开发效率
- 把复杂媒体逻辑和桌面 UI 解耦
- 方便后续做独立测试
- 未来可以逐步把能力从 CLI 下沉到原生层

## 7. 两条技术路线

## 7.1 路线 A：ffmpeg 产品化路线

这一条路线不追求自己实现编解码，而是把 `ffmpeg` 驱动到足够强。

重点能力：

- 自动字幕轨选择
- 复杂滤镜图生成
- GPU 编码和 CPU 回退
- 批处理规则
- 目录监听
- 自动入队
- 成品校验
- 错误诊断
- 预设系统
- 失败重试

适合优先做的原因：

- 见效快
- 风险低
- 维护成本低
- 已有工程基础
- 更容易先做成可用产品

## 7.2 路线 B：libav / C++ 引擎化路线

这一条路线是深入到底层控制开发，逐步从“调用 `ffmpeg`”升级为“构建自己的媒体引擎”。

典型能力：

- 自定义探测逻辑
- 自定义解封装和封装流程
- 自定义滤镜图管理
- 更细粒度的逐帧处理
- 更稳定的进度与错误回传
- 未来支持实时预览
- 未来支持截图分析和片段识别

这一条路线大概率需要：

- `C/C++`
- `Visual Studio C++ toolchain`
- `FFmpeg dev libraries`
- 更细的内存与资源管理

结论：

- 路线 A 先做产品
- 路线 B 作为第二阶段能力下沉路线

不要一开始把二者混成一坨。

## 9. 原生核心的发展目标

`native/AnimeMediaCore` 的长期目标不是“一口气替代全部 ffmpeg”，而是按收益逐步下沉。

推荐分三层下沉：

### 9.1 第一层：探测能力下沉

优先把下面这些逻辑下沉到原生模块：

- 媒体流分析
- 字幕轨候选信息归一化
- 语言和标题识别
- 输出校验辅助

原因：

- 这类逻辑相对独立
- 风险比全链路转码低
- 很适合先建立 `C# <-> DLL` 边界

### 9.2 第二层：处理能力下沉

逐步下沉这些能力：

- 自定义 filter graph 生成
- 截图抽帧
- 帧级统计
- 片段分析
- 预处理与后处理阶段

### 9.3 第三层：转码核心下沉

在确实有收益时再考虑：

- 自己控制解码
- 自己控制滤镜图
- 自己控制编码
- 自己控制封装

这一层复杂度最高，必须在前两层跑稳后再做。

## 10. C 与 C++ 的使用策略

建议策略：

- 对外导出接口尽量使用 `C`
- 内部实现使用 `C++`

原因：

- `C ABI` 更稳定
- 更容易被 `C# P/Invoke` 调用
- 更利于未来扩展到别的调用方
- 内部仍可使用 `C++` 的封装能力和 RAII

推荐做法：

- `include/` 放公开头文件
- `src/` 放 `.c` 与 `.cpp` 实现
- `tests/` 放原生模块测试

示例边界：

```c
int amc_probe_media(const char* input_path, char* output_json, int output_capacity);
int amc_validate_output(const char* input_path, const char* output_path, char* error_message, int error_capacity);
```

第一版先暴露“小而稳”的函数，不要一开始导出庞大接口。

## 13. 开发顺序建议

推荐顺序如下：

1. 先把现有 `WPF + ffmpeg CLI` 路线做稳
2. 做真实样本联调
3. 做错误处理、重试、历史记录
4. 做 `Release` 打包
5. 建立 `native/AnimeMediaCore` 基础工程
6. 先下沉探测与校验能力
7. 再评估是否值得下沉转码核心

## 14. 风险与注意事项

### 14.1 技术风险

- 不同字幕类型兼容性差异
- `NVENC` 在不同显卡和驱动版本上的行为差异
- ffmpeg 构建版本差异
- 中文路径和特殊字符路径处理
- 原生层调试难度显著高于 `C#`

### 14.2 产品风险

- 目标边界一旦放宽，会很快膨胀成“万能转码器”
- 过早进入 C++ 底层开发会拖慢可交付速度
- 如果没有明确工作流，底层投入很难转化成用户价值

### 14.3 工程原则

- 产品层与媒体层分离
- 先做稳定，再做深度
- 先做高频收益，再做技术挑战
- 对外接口小而稳

## 15. 当前结论

项目应该升级，但升级方式不是推翻当前 `WPF + ffmpeg` 方案，而是：

- 保留当前产品层
- 增加原生模块边界
- 先继续把 `ffmpeg` 路线做到成品级
- 再逐步把高价值能力下沉到 `C/C++`

这条路径的好处是：

- 现在就能继续交付可用功能
- 将来还能进入更深的媒体控制开发
- 不会因为过早下沉到底层而失去产品推进速度

## 16. 下一步建议

下一步建议直接进入两条并行工作：

- 产品线：继续完善 GUI、日志、重试、历史、目录监听、`Release` 发布
- 架构线：给 `native/AnimeMediaCore` 建立最小原生工程和第一个 `C ABI` 接口

这样项目既能继续“像软件”，也能开始具备真正的媒体底层演进空间。
