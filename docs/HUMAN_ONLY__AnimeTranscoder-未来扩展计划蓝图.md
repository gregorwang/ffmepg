# AnimeTranscoder 未来扩展计划蓝图

## 1. 文档目标

这份文档用于描述 `AnimeTranscoder` 在未来几个阶段中可以演进的方向、优先级、实施策略、技术边界与产品判断标准。

它并不是一个“想到什么写什么”的愿望清单，而是以当前项目已经具备的工程基础为前提，提出一套尽量可执行、尽量有先后顺序、尽量避免无效扩张的扩展蓝图。

文档的重点不是列功能名，而是回答以下问题：

- 这个项目接下来该先做什么
- 为什么先做这些，而不是别的
- 哪些功能能直接提升真实工作流价值
- 哪些扩展需要先补工程基础
- 怎样避免把项目扩成一个边界模糊的“万能媒体软件”

## 2. 未来扩展的总原则

在继续扩展项目前，必须先确定扩展原则，否则功能越多，产品价值反而越模糊。

### 2.1 工作流优先，而不是功能堆叠优先

未来扩展必须继续围绕真实工作流，而不是围绕“看起来很强”的参数列表。

应该优先做的，是能显著减少重复劳动、提高成功率、降低返工率的能力，例如：

- 自动兼容性判断
- 失败重试和恢复
- 质量巡检
- 批量规则化命名
- 预设模板

不应该优先做的，是那些增加大量 UI 复杂度、却只在少数场景偶尔使用的边角控制项。

### 2.2 默认策略要比高级选项更重要

这个项目未来要成为产品，而不是一个把所有 `ffmpeg` 选项原样搬到界面上的参数面板。

因此，未来每增加一个能力，都应该优先思考：

- 默认值是否足够安全
- 默认输出是否足够兼容
- 默认执行是否足够稳定

高级选项可以有，但不能反过来主导整个产品设计。

### 2.3 产品边界必须清晰

未来扩展时，项目要坚持以下边界判断：

- 可以做“围绕动漫本地处理的工具链”
- 不必做“替代所有视频软件的通用平台”

这意味着：

- 可以继续做整理、转码、巡检、抽帧、片段处理、竖屏适配、封面生成、字幕相关能力
- 不必追求做完整时间轴编辑器、调色软件、非线性剪辑系统

### 2.4 工程稳定性必须和功能同步升级

项目每增加一层媒体能力，就更不能只靠“能跑一次”来判断完成。

未来所有重要扩展都应同步考虑：

- 日志
- 错误分类
- 兼容性校验
- 存储预检
- 历史记录
- 最小自动化测试

## 3. 扩展路线总览

为了避免开发顺序混乱，未来扩展可以分成五个阶段：

- 第一阶段：稳定性与兼容性强化
- 第二阶段：批处理与预设系统强化
- 第三阶段：质量巡检与内容分析强化
- 第四阶段：二创与短视频工作流强化
- 第五阶段：原生媒体核心深化与工程平台化

下面分别展开。

## 4. 第一阶段：稳定性与兼容性强化

这是最应该优先完成的一阶段。

原因很简单：如果一个媒体工具不能稳定产出“能播、能验、能复跑”的文件，那么新功能越多，问题越多。

### 4.1 兼容策略体系化

当前已经完成了“默认 `AAC stereo`、加强音轨校验、空间预检”的第一步，但未来还可以做得更系统。

建议增加正式的输出兼容策略枚举，例如：

- `兼容优先`
- `均衡模式`
- `保留多声道`

对应行为可包括：

- 音频声道数
- 采样率
- 是否允许某些容器/参数组合
- 输出校验标准

这样做的好处是：

- 用户不需要理解所有底层参数
- 程序仍然可以提供更灵活的策略层
- 不同目标场景可以共享同一套 UI 逻辑

### 4.2 输出盘与临时空间管理

这次真实问题说明，磁盘空间不是附属问题，而是主流程稳定性的关键部分。

未来建议把存储管理升级到以下层级：

- 启动前检查目标盘空间
- 区分“正常写入需求”和“faststart 额外写入需求”
- 在队列界面显示当前输出盘剩余空间
- 对高风险任务给出黄色/红色预警
- 提供“建议切换输出目录”的引导

进一步的增强还可以包括：

- 允许指定临时工作盘
- 允许把 `faststart` 阶段写到另一块盘
- 允许使用“先本地写完，再移动到目标目录”的策略

### 4.3 失败分类与可恢复性

未来失败结果不应只停留在“任务失败”这一级，而应细分为：

- 输入文件缺失
- 媒体探测失败
- 字幕选择失败
- 编码器不可用
- 磁盘空间不足
- 输出容器不完整
- 输出兼容性不符合预期
- 用户取消

一旦失败分类明确，项目就可以继续做：

- 分类统计
- 分类重试
- 分类提示
- 历史报表

### 4.4 自动化验证

第一阶段还应补一组最小但高价值的自动验证。

建议优先覆盖：

- 主转码 builder 输出参数校验
- `ffprobe` 解析关键字段
- 输出校验逻辑
- 空间预检逻辑
- 文件已存在时的跳过逻辑

目标不是一开始就做到重度媒体测试，而是先把最容易回归的问题固定下来。

## 5. 第二阶段：批处理与预设系统强化

当稳定性到位后，最值得继续做的是“让批量工作流更省心”。

### 5.1 任务模板与预设系统

未来建议引入正式的预设系统，而不是只依赖若干分散设置字段。

预设可以分为两类：

- 输出预设
- 工作流预设

输出预设例如：

- 番剧归档兼容版
- 小体积移动端版
- 保留更多画质版
- 客厅播放兼容版

工作流预设例如：

- 监听目录自动转码
- 下载后自动烧录
- 竖屏二创前处理

预设的价值在于：

- 用户不需要重复调参数
- 项目可以把“经验”沉淀成产品能力
- 团队未来可以维护一套推荐策略

### 5.2 批量命名规则

未来建议把文件命名与输出目录规则做成可配置系统。

可能的命名能力包括：

- 保留原文件名
- 追加处理后缀
- 按番剧名/季/集拆目录
- 自动规范非法字符
- 自动缩短超长路径

在大量媒体归档场景里，命名规则的重要性往往不亚于编码参数。

### 5.3 更强的目录监听编排

当前已有监听能力，但未来可以继续增强：

- 多目录监听
- 不同目录套用不同预设
- 监听规则黑白名单
- 文件稳定时间自适应
- 夜间自动处理
- 失败文件暂停重试

这会让项目逐步接近“轻量本地媒体流水线”。

### 5.4 队列控制增强

未来可增加：

- 队列优先级
- 任务置顶/置底
- 暂停/恢复
- 失败后继续或终止策略
- 成功后自动关机/休眠
- 按时间窗口执行

这些能力都非常适合桌面批处理场景。

## 6. 第三阶段：质量巡检与内容分析强化

当前项目已经具备巡检雏形，这是非常值得继续深挖的一条线。

### 6.1 输出质量抽检体系

未来建议把当前抽帧巡检从“工具功能”升级成“质量体系”。

可以包括：

- 自动抽样首段、中段、尾段
- 自动检查字幕是否出现
- 自动检测黑屏、冻结、异常亮度
- 自动生成质量报告
- 自动标记需要人工复核的任务

一旦这部分成熟，项目就会明显超出普通转码 GUI 的层级。

### 6.2 字幕巡检

当前虽然能烧字幕，但未来还可以继续做：

- 判断字幕是否实际出现在画面中
- 判断字幕是否超出安全区域
- 判断字幕颜色/描边对比度是否过低
- 判断字幕轨选择是否疑似错误

这对于动漫场景尤其有价值，因为字幕烧录成功与否直接决定成品可用性。

### 6.3 封面候选与关键帧筛选

项目已经有封面候选能力，未来可以继续升级：

- 根据场景变化密度筛选关键帧
- 排除黑屏、模糊帧、字幕遮挡严重帧
- 自动评分推荐封面
- 支持为短视频导出封面图集

### 6.4 媒体内容结构分析

未来可以把内容分析继续向下做，例如：

- OP/ED 粗识别
- 静音段识别
- 黑场分段识别
- 高动势片段候选
- 字幕密集区识别

这会为后续“二创直出”“自动高光片段生成”打下基础。

## 7. 第四阶段：二创与短视频工作流强化

项目已经具备一定短视频链路，但未来还可以更系统化。

### 7.1 抖音模板系统

当前已有抖音直出模板，未来可以正式升级为模板引擎。

模板能力可以包括：

- 标题区布局
- 水印布局
- 字幕布局
- 背景模糊样式
- 主画面裁切策略
- BGM 混音规则
- 片头/片尾包装

这样项目就能从“会导出竖屏视频”进化成“可管理的一套模板化直出系统”。

### 7.2 自动高光片段流水线

这是未来非常有想象空间的一条路线。

可分解为：

- 检测场景切换
- 结合字幕密度与音量变化选候选片段
- 结合黑场与冻结检测做过滤
- 自动生成若干高光候选
- 一键导出为竖屏、GIF、封面

这条路线如果做好，会让项目从“处理素材”进一步升级为“生成内容”。

### 7.3 多平台导出

在保持边界清晰的前提下，未来可以扩展到多个短视频平台导出预设，例如：

- 抖音
- 快手
- B站竖屏
- YouTube Shorts

这里不需要一开始就做平台级集成，而可以先从分辨率、时长、音频规范、封面尺寸等输出规则做起。

### 7.4 片段级批处理

当前批量能力主要集中在整片处理，未来可以扩展到“片段批处理”：

- 批量拼接多个高光片段
- 批量导出 GIF
- 批量导出不同速度版本
- 批量做竖屏重构

## 8. 第五阶段：原生媒体核心深化与平台化

这是技术含量最高、但应当晚于前几个阶段的长期路线。

### 8.1 原生核心能力继续下沉

当前 `native/AnimeMediaCore` 已经承载了一部分探测与分析能力，未来可以继续下沉更多逻辑：

- 原生媒体探测统一入口
- 原生输出校验
- 原生帧采样
- 原生字幕样本分析
- 原生日志结构化输出

这样做的意义是：

- 减少对 CLI 输出文本解析的依赖
- 提高结构化能力
- 为后续更复杂分析提供更稳基础

### 8.2 原生层测试体系

如果原生层继续扩张，就必须建立独立测试体系。

建议逐步增加：

- 小样本媒体测试集
- 字幕探测测试
- 音轨探测测试
- 输出校验测试
- 抽帧一致性测试

### 8.3 原生层 ABI 稳定化

如果未来还有其他前端或其他调用方，原生层应尽早建立稳定 ABI 约定。

包括：

- 明确 C 导出接口
- 明确错误码体系
- 明确 JSON 输入输出协议
- 明确版本兼容策略

### 8.4 从单机工具到本地平台

如果项目继续扩展，未来甚至可以演化出“本地媒体处理平台”的雏形：

- 桌面 GUI 前端
- 原生媒体核心
- 本地任务服务
- 预设与模板仓库
- 可复用测试样本库

这一步不需要仓促推进，但从结构上看，当前项目已经具备向这一方向演化的基础。

## 9. 未来 UI/交互层建议

未来不只是加功能，还应提高可理解性。

建议的 UI 方向包括：

### 9.1 明确模式化入口

把界面操作从“大量按钮”逐步改造成“按工作流分类”：

- 主转码
- 音频处理
- 巡检分析
- 片段处理
- 竖屏直出
- 批量监听

### 9.2 重要风险前置提示

例如：

- 输出盘空间不足
- 当前策略可能降低兼容性
- 已存在目标文件但校验失败
- 当前任务将启用 `faststart`，收尾时间可能较长

### 9.3 结果摘要更可读

每次任务结束后，不只是显示成功或失败，而应形成更结构化摘要：

- 输入文件
- 输出文件
- 编码器
- 字幕来源
- 音频策略
- 校验结果
- 风险提示

## 10. 文档体系扩展建议

随着项目继续扩展，文档体系也应同步分层。

未来建议至少形成以下几类文档：

- 产品介绍文档
- 快速使用文档
- 输出兼容策略文档
- 预设系统文档
- 原生核心开发文档
- 测试样本与验证规范文档
- 常见故障与排查文档

文档体系越成熟，项目越容易从“作者个人工具”成长为“可协作项目”。

## 11. 建议的近期实施优先级

如果只看未来一到两个迭代，建议优先级如下：

### P0

- 完善兼容策略配置
- 完善空间预检和错误提示
- 补最小自动测试
- 统一所有 MP4 输出链路的兼容规则

### P1

- 增加预设系统
- 增加输出命名规则
- 增加队列暂停/恢复
- 增加更强的质量巡检摘要

### P2

- 增加自动高光候选
- 完善抖音模板系统
- 增强目录监听编排

### P3

- 深化原生媒体核心
- 建立更系统的测试样本库
- 向本地平台化架构推进

## 12. 结语

`AnimeTranscoder` 最值得继续投入的地方，在于它已经不是一张空白纸。

它已经有：

- 能跑通的主工作流
- 足够明确的产品定位
- 可以继续扩展的目录结构
- 已经开始体现差异化的功能集合
- 真实场景驱动出来的兼容性修正经验

未来只要坚持“工作流优先、默认策略优先、稳定性优先、边界清晰”的扩展原则，这个项目完全可以逐步成长为一套真正有辨识度的本地动漫媒体处理工具链。
