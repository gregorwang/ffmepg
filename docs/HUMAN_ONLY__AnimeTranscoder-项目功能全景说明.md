# AnimeTranscoder 项目功能全景说明

## 1. 文档定位

这份文档用于系统性介绍 `AnimeTranscoder` 当前已经具备的能力、核心工作流程、项目结构、关键模块职责、面向用户的实际价值，以及在真实本地媒体处理场景中的使用边界。

它不是一份“开发速查表”，也不是一份只写接口名的源码索引，而是一份面向以下几类读者的正式项目说明：

- 想快速理解项目用途和定位的使用者
- 需要接手项目继续维护的开发者
- 想判断这个工具能否承接个人工作流的压制用户
- 希望把当前原型继续产品化的人

如果只用一句话概括当前项目：

`AnimeTranscoder` 是一款面向番剧本地处理场景的 Windows 原生桌面工具，目标是把“探测媒体、选择字幕、烧录硬字幕、统一转码、批量执行、输出校验、历史追踪、基础巡检”这一整套高频重复流程，压缩成一条尽量稳定、尽量少手工参数、尽量可持续扩展的桌面工作流。

## 2. 项目目标与产品定位

### 2.1 项目目标

项目当前的核心目标非常明确，不是做一个万能的媒体后期平台，而是做一套围绕动漫视频处理的“高频场景工具”。

当前目标包括：

- 让用户可以以最低成本把 `mkv` 动漫视频整理为统一、可播放、可归档的成品
- 默认输出适合日常设备播放的 `MP4 + H.264 + AAC`
- 支持硬字幕烧录，避免外挂字幕在不同设备上的兼容问题
- 尽量自动识别合适字幕轨，减少手工挑选
- 优先使用 `NVENC` 等硬件能力，提高处理效率
- 用任务队列、目录监听、历史导出、输出校验等机制，把流程从“零散操作”变成“可复用工作流”

### 2.2 产品定位

这个项目最合适的定位不是“视频编辑器”，也不是“通用转码器”，而是“围绕固定媒体工作流的桌面产品”。

这意味着它更重视以下事情：

- 让默认参数合理
- 让批量处理稳定
- 让输出能播
- 让失败原因可见
- 让用户不必每次都重新思考复杂命令行

与专业视频软件相比，它不追求完整后期能力；与大量通用转码 GUI 相比，它更强调面向特定工作流的自动化与默认策略。

### 2.3 当前最适合的用户群

当前版本最适合以下类型的用户：

- 习惯本地保存番剧资源，并希望统一格式归档的人
- 需要把内嵌字幕内容烧录进视频的人
- 不想每次重复写 `ffmpeg` 命令的人
- 需要在 Windows 本地环境下连续处理多个文件的人
- 希望后续继续扩展项目，而不是把它视为一次性脚本的人

## 3. 当前项目已经实现的主要能力

项目当前并不是只有一个“转码按钮”的骨架，而是已经具备一套比较完整的桌面应用功能集合。下面按能力域进行说明。

### 3.1 主转码能力

主转码能力是项目当前的核心。

它已经具备以下特征：

- 从输入文件读取媒体信息
- 自动分析可用字幕轨
- 根据字幕偏好自动选择目标字幕流
- 通过 `ffmpeg subtitles` 滤镜烧录硬字幕
- 将视频编码为 `H.264`
- 将音频编码为 `AAC LC`
- 生成 `MP4` 封装输出
- 在硬件可用时优先使用 `h264_nvenc`
- 在无法使用硬件编码时回退到 CPU 路径

在当前版本中，这条链路不是单纯的“命令拼接”，而是被拆成多个职责明确的服务：

- `FfprobeService`
- `SubtitleSelectionService`
- `HardwareDetectionService`
- `FfmpegCommandBuilder`
- `FfmpegRunner`
- `OutputValidationService`

这意味着项目已经从“单文件脚本”走到了“可维护模块化程序”的阶段。

### 3.2 字幕相关能力

字幕能力是该项目区别于普通转码器的重要部分。

当前项目支持的字幕相关能力包括：

- 读取字幕流信息
- 识别字幕轨标题、语言、默认标记
- 根据简中/繁中偏好自动选取轨道
- 支持在缺少精确标签时回退到默认轨道或首轨
- 在 UI 中显示字幕分析来源与字幕摘要

这部分能力的价值并不只是“自动化一点点”，而是直接减少批处理时的人工决策负担。

在大量番剧资源的实际场景里，字幕流经常存在：

- 标题不规范
- 语言标签不完整
- 多条字幕并存
- 默认轨配置不一致

如果每个文件都要求用户手动点选字幕轨，这个工具的批量价值会大幅下降。因此字幕自动选择并不是附属功能，而是工作流自动化的关键环节。

### 3.3 媒体探测能力

项目当前通过 `ffprobe` 和原生媒体核心两种路径实现媒体探测。

探测结果包括但不限于：

- 总时长
- 音轨列表
- 字幕轨列表
- 音轨声道数
- 音轨采样率
- 音轨码率
- 字幕标题、语言和默认标记

在最新改进中，探测结果还进一步补充了：

- 文件大小
- 音轨 `channel_layout`

这一点很重要，因为它让项目不再只能判断“有没有音频”，还能进一步判断“这条音频是否足够兼容目标播放器”。

### 3.4 输出校验能力

输出校验是保证工具“不是只会生成文件，而是会生成合格文件”的关键。

旧式的简单转码工具经常在以下场景里失效：

- 进程退出了，但输出文件不完整
- 时长对上了，但容器元数据异常
- 文件存在，但播放器不认
- 中途失败后留下一半的残缺文件，后续误判为已完成

`AnimeTranscoder` 当前已经具备输出校验能力，并且这部分能力经过进一步加强：

- 先判断文件是否存在
- 再尝试读取输入输出媒体信息
- 计算输入输出时长差
- 判断差值是否在容差范围内
- 判断输出文件是否可被 `ffprobe` 正常解析
- 判断是否存在可用音视频流
- 判断音轨布局是否符合兼容性要求

也就是说，当前项目已经不再把“文件在磁盘上存在”视作“任务成功”的依据，而是引入了更接近成品质量的判定逻辑。

### 3.5 音频兼容性改进能力

这是本轮实际修复里非常关键的一部分。

在真实使用中，某些 `AAC 5.1` 或声道布局元数据异常的 `MP4` 文件，虽然可以被 `ffmpeg` 正常读取，但会被：

- Windows 原生播放器
- PS5 相册播放器
- 部分硬件解码设备

直接判定为不可播放。

为了解决这一类问题，项目现在已经在主输出链路中引入了明确的兼容优先策略：

- 默认音频输出为 `AAC stereo`
- 显式指定 `-ac 2`
- 显式指定 `-ar 48000`
- 输出校验阶段会识别音轨布局风险

这个调整的意义是：

- 项目的默认成品从“技术上合法”变成“更适合普通设备实际播放”
- 工具不再默认把保留源多声道视为优先目标
- 输出策略更贴合桌面归档和客厅设备播放场景

### 3.6 硬件检测与编码器选择能力

项目当前并不是硬编码某一种编码器，而是具备编码器选择和硬件回退策略。

主要流程为：

- 检测 `ffmpeg` 是否可用
- 检测 `NVENC` 能否使用
- 根据设置项和检测结果决定最终的视频编码器
- 在不可用时回退到 CPU 编码路径

这一能力的意义在于，它避免了用户必须知道底层硬件细节才敢启动任务。

对于工具产品来说，“自动决定合理实现路径”通常比“给用户暴露更多编码器选项”更重要。

### 3.7 任务队列能力

项目支持基于任务队列的串行执行。

当前已经具备：

- 文件加入队列
- 文件夹扫描入队
- 拖拽入队
- 队列状态切换
- 失败任务重试
- 完成任务清理
- 单任务状态查看
- 完成/失败/跳过/取消统计

这意味着项目并不是一次处理一个文件的“表面 GUI”，而是可以真正承担批量工作流的程序。

在大量文件处理中，队列机制的价值主要体现在：

- 统一控制执行顺序
- 统一展示任务状态
- 统一记录历史结果
- 允许用户在处理过程中观察失败点

### 3.8 输入目录监听能力

项目已经实现了输入目录监听与稳定文件自动入队能力。

这部分能力的场景意义很明显：

- 某个下载目录不断有新文件落地
- 工具持续监视该目录
- 当文件写入稳定后自动加入队列
- 可选自动开跑

这让项目从“用户主动点击执行”进化到“半自动工作流平台”的方向。

对常见的下载后处理场景来说，这一步实际上非常有价值，因为它减少了人为触发步骤，提升了持续处理能力。

### 3.9 日志与历史能力

当前项目提供两层记录机制：

- 运行日志
- 历史任务记录

日志用于追踪执行过程，历史记录用于沉淀结果。

历史能力包括：

- 状态筛选
- 日期筛选
- `json/txt` 导出
- 队列结束统计

这部分能力的价值体现在：

- 处理大批量文件后可以回溯结果
- 用户可以知道哪些任务被跳过、哪些任务失败
- 开发者可以基于日志定位错误来源

### 3.10 诊断抽帧与巡检能力

项目当前不仅支持转码，还已经具备初步的媒体质量巡检能力。

当前提供：

- 单帧诊断截图
- 批量抽帧
- 巡检三连图
- 亮度与对比度分析
- 巡检报告导出

虽然这部分能力还处于“巡检辅助”阶段，但它已经为未来进一步做自动化质量判断打下了基础。

在番剧处理工作流中，这类能力可用于：

- 快速判断字幕是否烧录成功
- 快速观察画面是否异常
- 对成片进行抽样巡检

### 3.11 音频处理能力

除了主转码功能，项目还具备独立的音频处理能力。

包括：

- 音频提取
- 音频格式导出
- 静音检测
- 背景音乐混音
- 归一化处理

这说明项目已经不是“只能做主任务”的单功能程序，而是在向更完整的媒体处理工具链延伸。

### 3.12 视频片段处理能力

当前项目还包含一组围绕短视频和二次创作的片段能力：

- 快速裁剪
- 精确裁剪
- GIF 预览导出
- 变速
- 倒放
- 画中画
- 片段拼接
- 竖屏适配
- 场景检测
- 黑场检测
- 冻帧检测
- 封面候选生成

这些能力共同说明，项目已经不仅仅是在做“番剧整理器”，而是正在逐步发展出一套围绕番剧素材利用的工具链。

### 3.13 抖音直出能力

项目还包含面向竖屏内容的“抖音直出”链路。

主要能力包括：

- 根据模板生成竖屏输出
- 叠加标题和水印
- 可选字幕烧录
- 可混入 BGM
- 统一输出 `mp4`

这部分能力使项目从“归档工具”延伸到“二次分发工具”，具有明显的产品扩展空间。

## 4. 当前核心模块结构

为了让后续维护更容易，项目已经按职责拆分成多层目录。

### 4.1 `Models/`

用于承载数据模型。

这里定义了：

- 任务对象
- 设置对象
- 探测结果
- 输出校验结果
- 音轨与字幕轨信息
- 各类音视频处理结果对象

这层的意义是统一不同服务之间的数据交换结构，让 ViewModel 和 Service 不直接共享零散字典或匿名数据。

### 4.2 `Services/`

这是当前业务能力最集中的目录。

核心服务包括：

- `FfprobeService`
- `FfmpegRunner`
- `FfmpegCommandBuilder`
- `OutputValidationService`
- `SubtitleSelectionService`
- `HardwareDetectionService`
- `DirectoryWatchService`
- `FrameInspectionService`
- `AudioExtractionService`
- `VideoClipService`
- `DouyinExportService`
- `StoragePreflightService`

它们的共同特征是：

- 每个服务只关注一类能力
- 尽量保持与 UI 解耦
- 方便后续替换底层实现

### 4.3 `ViewModels/`

`MainViewModel` 承担当前主界面的状态组织与命令编排。

它负责：

- 绑定 UI 显示状态
- 接收用户操作
- 调用服务执行任务
- 更新任务队列
- 维护历史和摘要信息

目前它仍然是体量较大的总控类，但已经能够说明整个项目工作流的主线。

### 4.4 `Views/`

负责 WPF 界面层。

当前主界面通过数据绑定接入 `MainViewModel`，承担：

- 参数输入
- 状态展示
- 文件拖拽
- 命令按钮
- 任务详情

### 4.5 `Infrastructure/`

这里放置偏通用、偏支撑的基础设施能力，例如：

- 命令封装
- 工具路径解析
- 文件日志记录

它的作用是把通用性代码从业务层剥离出来，降低耦合。

### 4.6 `native/AnimeMediaCore/`

这是项目向原生媒体核心发展的重要边界。

当前它已经包含：

- 原生媒体探测
- 原生字幕分析
- 原生输出校验
- 原生抽帧接口

它的重要性不在于当前已经替代所有 CLI，而在于它定义了未来继续深入底层媒体逻辑的入口。

## 5. 当前完整工作流说明

从用户点击开始到最终产出成品，当前项目的主链路大致如下：

1. 用户通过拖拽、文件选择或目录扫描加入输入文件
2. 系统为每个任务生成输出路径
3. 若关闭覆盖模式，则先检查目标文件是否已经存在且有效
4. 在最新版本中，转码前会先检查输出盘空间是否足够
5. 系统探测媒体信息，包括时长、音轨、字幕轨
6. 系统分析字幕轨并根据偏好选择目标字幕
7. 系统检测硬件编码可用性并确定编码器
8. 构建 `ffmpeg` 参数并启动转码
9. 实时回传进度与速度信息
10. 转码完成后执行输出校验
11. 校验通过则任务记为成功，否则记为失败
12. 按配置决定是否删除源文件
13. 将结果写入历史记录

这一流程里最关键的设计思想是：

- 所有关键步骤都变成显式阶段
- 失败可以尽量落在清晰的阶段上
- 用户可以通过日志知道当前正在做什么

## 6. 当前版本解决了哪些真实问题

项目目前已经解决了若干真实场景中的痛点。

### 6.1 解决“每次都要重新写命令”的问题

对于有固定工作流的用户来说，命令行并不难，真正烦的是重复。

这个项目把高频重复命令产品化后，价值主要体现在：

- 降低重复劳动
- 降低记忆成本
- 降低参数遗漏概率

### 6.2 解决“字幕设备兼容差”的问题

外挂字幕在设备间兼容性非常差，而硬字幕输出是最直接的解决方式。

### 6.3 解决“批量处理不稳定”的问题

单独执行命令时，用户往往只能盯着一个终端窗口。

项目通过队列、历史、日志、跳过逻辑，把批处理变成了可跟踪的过程。

### 6.4 解决“文件看起来生成了，但其实不能播”的问题

这是这次实战里最重要的经验之一。

当前版本通过以下改进提升了这一点：

- 默认输出更偏兼容
- 音轨布局被明确控制
- 输出校验不再只看时长
- 失败时能识别 `No space left on device`
- 预检会尽量在启动前发现磁盘不足

### 6.5 解决“faststart 在慢盘上拖很久，用户误以为卡死”的问题

这部分还没有变成完整的 UI 提示体系，但当前已经做了两件非常重要的事：

- 把 `faststart` 是否启用变成配置项
- 在文档和日志中明确它会带来额外 I/O 成本

## 7. 当前版本的兼容性策略

在最新策略下，主转码输出默认面向“普通本地播放兼容优先”。

主要规则为：

- 容器：`MP4`
- 视频：`H.264`
- 音频：`AAC LC`
- 声道：默认 `stereo`
- 采样率：`48000`
- 像素格式：`yuv420p`

这样设计的原因不是因为它“技术上最先进”，而是因为它在下面这些目标之间取得了更实用的平衡：

- Windows 原生播放器兼容性
- 客厅设备兼容性
- 通用硬件解码兼容性
- 文件体积和质量平衡
- 批处理稳定性

## 8. 当前的工程价值

从工程角度看，当前项目已经具备几个值得继续投入的特征：

- 已经模块化，而不是一次性脚本
- 已经有可扩展目录结构
- 已经接入原生能力边界
- 已经具备多条媒体处理链路
- 已经有日志、历史、巡检、导出等产品化能力
- 已经开始从“能跑”转向“能稳定产出”

这意味着后续继续做增强时，不需要推翻现有工程，而是可以围绕当前结构持续演进。

## 9. 当前已知边界与限制

尽管当前项目已经有较多能力，但它依然存在清晰边界：

- 主转码目标仍然是固定工作流，不是任意媒体流程编辑器
- 主界面总控逻辑仍较集中，后续还可以继续拆分
- 兼容策略当前以“通用设备播放优先”为主，不是影音发烧场景优先
- `faststart` 的成本虽已被识别，但仍有进一步优化空间
- 当前没有系统化自动测试来覆盖所有媒体组合

这些限制并不意味着项目价值不足，而是意味着它已经走到一个可以清晰继续产品化的阶段。

## 10. 结语

`AnimeTranscoder` 当前最重要的意义，不是“已经做成了一个完整商业软件”，而是它已经形成了真实、清晰、可持续扩展的产品雏形。

它已经完成了从“脚本堆叠”到“桌面应用工作流”的跨越，也已经开始处理真正重要的问题：

- 输出兼容性
- 批量稳定性
- 可观察性
- 失败可诊断性
- 未来可扩展性

如果继续沿着当前路线推进，这个项目完全有机会成长为一套面向番剧本地处理、剪辑辅助和内容直出的一体化 Windows 工具链。
