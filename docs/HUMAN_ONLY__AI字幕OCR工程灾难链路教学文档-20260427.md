# AI 字幕 OCR 工程灾难链路教学文档

生成日期：2026-04-27

适用项目：`AnimeTranscoder` / `GameSubtitleOCR` / `Ghost Yotei` 双语字幕工作流

## 0. 文档定位

这份文档回答两个问题。

第一，解释几个术语：

- 批处理速度是什么意思。
- 是否能跑视觉语言模型是什么意思。
- 是否能做多帧融合是什么意思。

第二，复盘一个更大的问题：

> 为什么一个 AI 辅助工程项目里，只要早期一步没有做好，后面就可能变成一条越来越昂贵、越来越难纠正的错误链？

本文会结合仓库中已经落地的文档和代码，不假装能读取当前会话之外的聊天记录。所谓“跨聊天记录提取细节”，在本文中具体指：从 `docs/` 里已经保存的历史复盘、交接文档、Phase B / Phase C 说明、`GameSubtitleOCR` 代码和项目 README 中提取事实。

本文不是给最终用户看的宣传稿，而是给操作者、下一轮 AI、未来的自己看的工程教学文档。它的语气会更直接，因为这个项目真正暴露的问题不是“某个命令没跑好”，而是“人、AI、数据、工程质量门之间的责任边界没有一开始就被定义清楚”。

## 1. 三个术语到底是什么意思

### 1.1 批处理速度

批处理速度不是单纯问“程序快不快”。

在这个项目里，它至少包含五层意思。

第一层是媒体处理吞吐量：例如一段视频每秒能抽多少帧、裁剪多少帧、预处理多少帧、送进 OCR 多少帧。`GameSubtitleOCR` 当前的全量抽取逻辑是按 `fps` 从视频流式取帧，再对每一帧裁剪、预处理、OCR、记录结果。默认参数里存在 `--fps 3.0` 这种抽样频率。也就是说，它不是逐帧 30fps 或 60fps 全量识别，而是以较低 FPS 采样来换取可承受的速度。

第二层是 OCR 推理吞吐量：同样一批帧，用 CPU、6GB 显存 GPU、12GB 显存 GPU、云端 GPU 跑，速度和可用模型大小完全不同。轻量模型可以跑得动，但识别上限较低；大模型可能更准，但显存不够就无法稳定运行，或者慢到不适合全片。

第三层是任务规模吞吐量：一个 3 分钟样片、一个 30 分钟视频、四个 part、五千多条英文 cue、四千多条中文 cue，完全不是同一个数量级。一个方法在样片上“看起来可以”，不代表能在全量项目里承受。

第四层是重跑成本：字幕项目不可能只跑一次。OCR 参数要调，裁剪框要改，低置信行要复核，时间轴要导出，硬字幕要烧录，抽帧 QA 要重跑。如果一次全量 OCR 要几个小时，每一次发现问题都要从头返工，那么批处理速度低就会直接拖垮工程节奏。

第五层是自动化调度能力：真正有价值的批处理不是手动点十次按钮，而是能用 CLI、脚本、manifest、指针文件、版本目录连续执行。仓库里后来已经形成 `AnimeTranscoder.Core`、`AnimeTranscoder.Cli`、`scratch/PHASE_C_CURRENT_MODEL_APPLIED.txt` 这类结构，说明项目已经从纯 GUI 操作走向可脚本化工作流。批处理速度因此也包括“能不能无人工守着跑完一批任务”。

所以，当有人问“批处理速度够不够”时，真正的问题是：

> 这条工作流能不能在可接受时间内，对全量视频、全量字幕、全量 QA 反复执行，而不是只在一个样片上演示成功？

如果答案是否定的，那么它就不能作为主生产流程，只能作为实验工具。

### 1.2 是否能跑视觉语言模型

视觉语言模型，通常指能同时处理图像和文字的模型。它不只是 OCR，也不只是语言模型。

普通 OCR 的输入是图片，输出是文字。它主要回答：

> 这张图上有哪些字？

语言模型的输入通常是文字，输出是文字。它主要回答：

> 这段文字是什么意思？是否对应？能否修正？

视觉语言模型可以把截图、字幕文本、上下文一起看。它可以回答更复杂的问题：

- 当前截图里的中文字幕有没有遮住英文硬字幕？
- 画面中的英文硬字幕是什么？
- 新烧录的中文字幕和画面里的英文字幕语义是否对应？
- 当前字幕像不像上一句或下一句？
- 这帧里是不是 UI 文本，不是对白字幕？
- 字幕位置、字号、描边是否影响观看？

这类能力对 Ghost Yotei 项目非常有价值，因为最终问题不是单纯“有没有中文文本”，而是“观众在某一秒看到的画面、英文硬字幕、中文字幕是否一致”。

但“能不能跑视觉语言模型”不是一句抽象判断，而是一个现实算力问题。

本项目历史文档里已经记录过一个重要限制：本机显存只有 6GB。这会影响：

- 能否本地运行较大的 VLM。
- 能否以足够分辨率输入截图。
- 能否批量处理几千张 QA 截图。
- 能否同时处理多帧上下文。
- 推理速度是否足够支撑全片检查。

如果本地跑不了，就要考虑云端模型。但云端模型又带来成本、速率限制、隐私、文件上传、结果复现的问题。

因此，这个问题真正的意思是：

> 我们是否具备一种可规模化、可重复、成本可接受的“看画面理解字幕”的模型检查能力？

如果没有，那么工作流就不能依赖它做全量质量保证，只能把它用在抽样 QA、低置信行复核、疑难截图判断上。

### 1.3 是否能做多帧融合

多帧融合不是简单地“从视频里多抽几张图”。

当前 `GameSubtitleOCR` 已经做了帧间处理：它按固定 FPS 抽取多帧，对每帧 OCR，然后用相似度把连续帧上的相近文本合并成一个字幕 cue。这是一种时间维度的合并，能减少重复字幕，能把同一句字幕的显示时间估出来。

但严格意义上的多帧融合还可以更进一步。

第一种是多帧投票：同一句字幕会在屏幕上停留若干帧。假设三帧 OCR 结果分别是：

- `我明天杀四他`
- `我明天杀死他`
- `我明天杀死他`

多帧投票应该选择 `杀死`，而不是只看某一帧的结果。

第二种是字符级融合：不是只比较整句，而是在字符位置上综合多帧结果。例如某一帧把 `死` 看成 `四`，另一帧把它看对了，系统可以根据置信度和重复次数做字符级修正。

第三种是图像级融合：对同一句字幕的多帧裁剪区域做对齐、去噪、锐化或超分辨率重建，再送 OCR。低码率视频里，单帧可能糊，但多帧可以提供更多信息。

第四种是时序一致性融合：字幕通常不会每 0.1 秒随机变化。如果连续几帧只有一个字不同，大概率是 OCR 抖动，不是真实字幕变化。系统应该用时序稳定性压制抖动。

第五种是多模型融合：同一帧或同一句字幕可以用不同 OCR 配置、不同预处理、不同模型识别，然后根据置信度和规则选择结果。

所以，“是否能做多帧融合”真正问的是：

> 我们能不能利用同一句字幕在多个连续帧中重复出现这一事实，把单帧 OCR 的错误率降下来？

这件事对低清视频尤其关键。因为低清画质下，单帧 OCR 是不稳定的；如果不用多帧信息，后面的语言模型就会吃到大量脏文本。

## 2. 这三个问题为什么是一个整体

批处理速度、视觉语言模型、多帧融合不是三个孤立功能。它们其实对应一条 OCR 字幕工程链的三个核心约束。

批处理速度回答：

> 能不能跑得动全量？

视觉语言模型回答：

> 能不能看画面做语义和版面判断？

多帧融合回答：

> 能不能在源头降低 OCR 噪声？

如果批处理速度不够，多帧融合和 VLM QA 就很难全量使用，只能抽样。

如果没有 VLM，系统很难判断“字幕是否真的和画面一致”，只能看文本。

如果没有多帧融合，OCR 噪声会进入后续所有匹配、翻译、时间轴、质量门。

这也是为什么它们会一起出现。它们共同决定这个项目是一个“能跑通的实验”，还是一个“能反复生产可靠结果的工具链”。

## 3. 本项目已经发生过什么

根据仓库文档，Ghost Yotei 项目大致经历了下面这些阶段。

最初目标不是普通转码，而是做中英双语字幕硬字幕版本。具体包括：

- 从英文硬字幕得到英文 cue。
- 从中文硬字幕或中文片段得到中文 cue。
- 把中文挂到英文时间轴上。
- 做模型校对和规则清洗。
- 导出 SRT。
- 用 ffmpeg 烧录到视频。
- 抽帧检查字幕位置。
- 生成发布材料。

中途曾考虑过 Whisper 路线。Whisper 可以从音频转写英文，但后来路线修正为英文也走 OCR。这个修正很关键，因为中文来自画面硬字幕 OCR，如果英文来自语音转写，两边的数据层级不同：一边是声音，一边是画面字幕。用语音时间线去匹配画面字幕时间线，会增加额外误差。

Phase B 的核心问题是：中文不是天然对应四个英文 part 的完整轨道，而是来自多个中文切片。因此要先解决 `13 clips -> 4 parts` 的坐标恢复问题。这个阶段不是翻译问题，而是“中文片段属于哪个英文 part、哪个大致区间”的定位问题。

Phase C 的核心问题是：以英文 OCR cue 为主轴，把中文 OCR cue 尽可能挂上去。历史文档记录过英文 OCR cleaned 总量约 `5158`，中文 OCR cleaned 总量约 `4423`。后续最终版本 `phase_c_model_applied_v36` 保留 `5153` 条英文 cue，`matched_cues = 5153`，`coverage_ratio = 1.0`，删除 `3` 条不可用行。

但这里最重要的教训是：

> `coverage_ratio = 1.0` 只能说明每条保留英文 cue 都有中文字段，不代表每句字幕都正确。

最终硬字幕输出 `scratch/phase_c_hardsub_v36` 已经跑通，字幕位置也经过抽帧检查。但历史复盘已经明确说过：这个产物更适合标为 AI 工作流实验 Beta，不应该宣称为严格精校成品。

## 4. “指数级灾难”这个直觉基本是对的，但要更精确

你说“只要有一步没做好，就会带来指数级灾难”，这个直觉是对的，但工程上可以说得更精确。

它未必是数学意义上的指数增长，而是多阶段系统里的误差放大。

假设一条流程有这些阶段：

1. 视频源质量。
2. 抽帧。
3. 字幕区域裁剪。
4. 图像预处理。
5. OCR。
6. OCR 清洗。
7. 中英片段定位。
8. cue 级匹配。
9. 模型校对。
10. SRT 导出。
11. 硬字幕烧录。
12. 抽帧 QA。
13. 发布声明。

每一层都依赖上一层的输出。如果上游错了，下游并不会自动知道“这是错的”。下游常常会把上游错误当成事实，然后继续加工。

一个低清视频导致 OCR 错字，这只是第一层问题。

OCR 错字进入匹配系统后，会降低语义相似度，这是第二层问题。

语义相似度降低后，系统可能选错中文候选，这是第三层问题。

选错中文候选后，模型可能为了“补齐覆盖率”生成看起来通顺的中文，这是第四层问题。

生成的中文进入 SRT 后，硬字幕烧录仍然会成功，这是第五层问题。

ffmpeg 成功退出后，如果只看文件存在和时长一致，系统会误以为完成，这是第六层问题。

最后观众看到的是“字幕有、位置对、但内容错”，这就是最危险的状态。

空字幕很容易被发现。错字幕反而更危险，因为它长得像成品。

这就是你说的“灾难”。它不是某个工具崩溃，而是整条链路都在认真处理一个已经污染的数据源。

## 5. 糟糕画质为什么会伤到后面每一步

字幕 OCR 不是抽象文本任务。它的第一输入是画面。

低清视频、二压视频、小字号字幕、运动模糊、复杂背景、描边阴影、UI 叠层，都会让 OCR 看到的不是“文字”，而是一团形状接近文字的像素。

这会带来几类错误。

第一类是可恢复错字。例如 `杀四他` 大概率可以根据上下文修成 `杀死他`。这类错误语言模型能帮忙。

第二类是专名错误。例如人名、地名、组织名识别错。专名一错，后续匹配会明显受影响，因为专名往往是锚点。

第三类是漏句。整句字幕没识别出来，这不是修几个字能解决的。

第四类是粘句。两句或多句被 OCR 合成一句，导致 cue 粒度错误。

第五类是 UI 混入。任务提示、按键提示、地点提示被当成对白字幕。

第六类是时间边界抖动。同一句字幕在连续帧中时有时无，导致 start/end 不稳定。

这些错误对后续阶段的破坏程度不同。可恢复错字只是文本清洗问题，漏句和粘句是结构问题，UI 混入是分类问题，时间抖动是时间轴问题。

最严重的是结构问题。因为一旦句子数量、边界、顺序错了，后面的中英匹配就不是“翻译判断”，而是在错误坐标系里找答案。

这就是为什么“给我 4K 源视频再 OCR”不是任性要求，而是工程上的上游质量控制。

如果源视频只有低清版本，项目不是不能做，但要降低承诺：

- 可以做实验。
- 可以做学习参考。
- 可以做局部高置信字幕。
- 不应该承诺全量精校字幕。

## 6. AI 为什么不会自动“摊牌”

你问的核心不是技术，而是 AI 协作的责任边界。

你想象中的场景是：AI 在项目中途意识到“继续做下去没意义”，然后直接告诉人类：除非提供更高清源，否则 OCR 根基不稳，后面做出来都是垃圾。

现实里，普通指令遵循模型通常不会这样工作，原因有几个。

第一，AI 的默认目标是完成用户当前请求，而不是主动重定义项目目标。用户说“继续处理”，AI 会倾向于找一条可继续执行的路径。除非提示词、系统约束或工程流程明确要求它在质量不达标时停止，否则它会继续推进。

第二，AI 没有稳定的项目风险预算。人类项目经理会问：这个错误率是否超过可接受阈值？继续投入是否值得？是否应该停机复盘？普通模型不会天然维护一个严格的“停工条件列表”。

第三，AI 很容易把“能生成产物”误当成“任务有进展”。在字幕项目里，生成 `all_segments.json`、导出 `zh.srt`、烧录 `mp4` 都是可见进展。但这些进展不自动等于质量正确。

第四，AI 的上下文有限。长项目跨多轮会话时，如果关键风险没有写进文档、manifest、质量门，AI 可能在新会话里只看到“当前版本 v36 覆盖率 100%”，而看不到“这只是结构覆盖，不是精校质量”。

第五，AI 不承担发布后果。人类上传视频后，观众看到错字幕，责任和羞耻感在人类这里。AI 没有这种社会反馈闭环。

所以，与其期待 AI 自动产生“多重思考意识”，更可靠的做法是把停机条件工程化。

例如，在项目开始时写死：

- OCR 抽样 100 条，错误率超过 10%，禁止进入全量匹配。
- 中英匹配抽样 50 条，明显错配超过 5 条，禁止导出最终 SRT。
- 低置信行超过 20%，最终版本只能标 beta，不能标 final。
- 没有抽帧 QA，不允许声明硬字幕完成。
- 画质低于某阈值，必须先尝试更高清源或降级目标。

这样 AI 不需要“觉醒”，它只需要执行质量门。

## 7. 人为什么也很难全局把控

你说“愚蠢的人类也就是我直到做完了才意识到问题在哪，我并没有对全局的把控能力，我也不知道什么时候会结束”。这不是愚蠢，而是复杂工程的正常失败模式。

因为人类很容易被局部进展牵引。

今天 OCR 跑通了，感觉前进了。

明天匹配表生成了，感觉又前进了。

后天覆盖率到 100%，感觉快结束了。

再后来 SRT 导出了，ffmpeg 烧录成功了，视频也能播放了，感觉已经完成了。

直到真正观看时，才发现字幕在关键处错了。

这不是单纯粗心，而是指标错位。

你看到的是“流程完成指标”：

- 文件生成了。
- 命令成功了。
- 行数对了。
- 覆盖率满了。
- 视频能播。

但真正需要的是“交付质量指标”：

- 字幕是否对应当前画面？
- 时间轴是否正确？
- OCR 是否可靠？
- 专名是否一致？
- 低置信行是否被标记？
- 抽样错误率是否可接受？
- 观众是否会被误导？

复杂项目最危险的地方就是：流程完成指标很容易自动化，交付质量指标很难自动化。

所以你不是缺“努力”，而是缺一个在流程早期强迫你面对坏消息的系统。

## 8. 覆盖率 100% 为什么尤其危险

`coverage_ratio = 1.0` 是这类项目里最容易误导人的数字。

它回答的是：

> 每条英文 cue 有没有中文字段？

它不回答：

- 中文是否真的对应英文。
- 中文是否来自官方 OCR。
- 中文是否是模型生成的。
- 中文是否时间上提前或滞后。
- 中文是否是上一句或下一句。
- 中文是否只是为了填满而补出来。

一个错误例子：

```text
英文：You are my brother.
中文：风从山谷里吹过。
状态：matched
覆盖率：已覆盖
```

从数据库角度看，这一行不为空，所以覆盖率提高了。从观看角度看，这一行完全错误。

因此，未来不应该只看覆盖率，而应该看分层指标：

- 高置信 OCR 原文匹配。
- 中置信语义匹配。
- OCR 可恢复修正。
- 模型生成补文。
- 无可用中文。
- 删除行。
- 低置信保留行。
- 抽样错配率。
- 时间轴抽样错误率。

如果最终版本里大量行来自 `model-text-fill` 或低置信匹配，那么它可以是学习版，但不能当精校版。

## 9. 时间轴问题的本质

历史 Phase B 文档里已经把时间轴拆成四层，这个拆法非常重要。

层级 A：单边 OCR 时间轴。英文 OCR 有 start/end，中文 OCR 也有 start/end。

层级 B：局部双语对齐时间轴。英文主轴上有一批局部连续段可以挂中文。

层级 C：大范围连续双语时间轴。多个 part 内出现较长连续双语挂载。

层级 D：完整成品级双语时间轴。覆盖全部 part，大部分 cue 稳定，误配率低，边界自然，可以交付。

本项目早期真正困难的地方不是“完全没有时间”，而是“有时间但不可信”。

英文 OCR 有时间，中文 OCR 也有时间，但它们不是同一条轨道上的双语字幕。英文是四个长 part，中文是多个切片。必须先恢复坐标系，再谈 cue 对齐。

而且中英文切分粒度不一致：

- 一句英文可能对应一句中文。
- 两句英文可能合成一句中文。
- 一句英文可能拆成两句中文。
- 英文 block 可能包含多句，中文 cue 更细。

这意味着对齐不是表格 join，而是带噪声的序列匹配。

时间轴错位比翻译错更伤观看体验。翻译稍微不自然，观众还能理解；字幕提前一秒或滞后一秒，会直接破坏观看。

所以未来 QA 不能只看文本，还要看截图和短片段。

## 10. AI 在这个项目里真正适合做什么

AI 很有用，但要放在合适的位置。

AI 适合做：

- 根据上下文判断 OCR 错字是否可恢复。
- 判断中文候选和英文是否大致对应。
- 找明显错配、重复、空字幕、英文残留。
- 给低置信行解释原因。
- 写脚本处理 JSON、SRT、TSV。
- 生成 QA 报告。
- 根据截图判断字幕是否遮挡。
- 帮助形成文档和交接说明。

AI 不适合单独承担：

- 从严重低清画面中恢复真实字幕。
- 在无锚点情况下对齐几千行双语 cue。
- 保证每一秒时间轴准确。
- 在没有质量门的情况下自动宣称成品完成。
- 用覆盖率替代观看质量。

最准确的说法是：

> AI 可以放大人的工程执行力，但不能替代工程质量制度。

没有质量制度时，AI 会让错误更快地走到终点。

## 11. 下一次应该怎样设计工作流

如果未来重做 Ghost Yotei 或类似项目，不建议一上来就全片。

### 11.1 先做 3 分钟样片

样片要故意包含复杂情况：

- 明亮场景。
- 暗色场景。
- 快速对白。
- 专名。
- UI 叠层。
- 多行字幕。
- 人物说话切换。

样片要跑完整链路：

1. 抽帧。
2. 字幕区域裁剪。
3. OCR。
4. 多帧合并。
5. 文本清洗。
6. 中英匹配。
7. SRT 导出。
8. 硬字幕烧录。
9. 抽帧 QA。
10. 错误率报告。

如果样片质量不过关，不允许进入全片。

### 11.2 OCR 独立验收

不要等最终视频出来才判断 OCR 好不好。

OCR 阶段应该单独输出：

- 原始帧截图。
- 裁剪框。
- OCR 原文。
- 置信度。
- 多帧候选。
- 最终合并文本。
- 随机抽样报告。

建议至少抽样：

- 英文 100 条。
- 中文 100 条。
- 每类错误统计。

如果 OCR 抽样已经严重不稳，后面不应该继续追求全量精校。

### 11.3 建立专名词表

游戏字幕里专名非常重要。人名、地名、武器名、组织名常常是匹配锚点。

未来应该先建立词表：

- 英文专名。
- 中文对应。
- 常见 OCR 错形。
- 是否允许保留英文。

例如 `Matsumae` 这类词，不能每次都让模型临场猜。

### 11.4 高置信锚点优先

先找少量一定正确的锚点，再做局部扩展。

不要一开始全局硬匹配五千行。

高置信锚点可以来自：

- 独特长句。
- 稀有人名。
- 地名。
- 连续对话段。
- 数字。
- 事件关键词。

一个 30 条但都正确的锚点集合，比 3000 条真假混杂的匹配表更有价值。

### 11.5 低置信保留为空，不要强填

未来系统应该允许“不知道”。

如果一条英文没有可靠中文，可以：

- 留空。
- 标记 `unmatched`。
- 标记 `needs_review`。
- 标记 `generated_translation`。

不要为了覆盖率把它强行填满。强填会让错误隐形。

### 11.6 每一步都要保留版本

本项目做得对的一点是保留了大量版本目录和指针文件。

未来应该继续坚持：

- 不覆盖原始 OCR。
- 不覆盖上一个 applied 版本。
- 每轮输出有 manifest。
- 当前版本用指针文件标识。
- 删除行有 deletion log。
- 模型改写有来源字段。

如果只保留一个 `final.json`，项目一旦出错就无法定位。

## 12. 一个更成熟的质量门设计

下面是一套更实用的质量门。

### 12.1 输入质量门

在 OCR 前判断：

- 视频分辨率是否足够。
- 字幕高度是否足够。
- 是否存在二压模糊。
- 字幕区域是否稳定。
- 是否有严重 UI 干扰。

如果输入质量明显不足，直接降级目标：只做实验，不做精校承诺。

### 12.2 OCR 质量门

OCR 抽样后判断：

- 字符错误率。
- 漏句率。
- 粘句率。
- UI 混入率。
- 专名错误率。
- 低置信占比。

如果漏句和粘句严重，不允许进入全量匹配。

### 12.3 匹配质量门

匹配后判断：

- 高置信比例。
- 低置信比例。
- 一对多 / 多对一比例。
- 专名冲突数量。
- 前后文不连续数量。
- 明显错配抽样比例。

如果低置信比例过高，不能用 `coverage_ratio` 掩盖。

### 12.4 时间轴质量门

时间轴导出后判断：

- 是否时间倒序。
- 是否字幕重叠异常。
- 是否持续时间过短或过长。
- 是否大段空洞。
- 是否连续几句都像错位。

时间轴必须结合截图或短视频片段验证。

### 12.5 烧录质量门

硬字幕后判断：

- 中文字幕是否遮挡英文硬字幕。
- 字号是否可读。
- 描边是否足够。
- 明暗场景是否都能看清。
- 是否乱码。
- 输出时长是否和源视频一致。

仓库现有 `AnimeTranscoder` 已经有抽帧巡检、输出校验、批量抽帧这些基础，可以继续扩展成正式 QA。

### 12.6 发布质量门

发布前判断：

- 是否能称为 final。
- 是否只能称为 beta。
- 是否需要声明 OCR 错误风险。
- 是否需要声明时间轴风险。
- 是否需要声明非完整精校。

本项目历史文档已经建议用 `Codex 工作流 AI 试验 Beta` 这类表述，这是诚实的。

## 13. 针对当前仓库的技术判断

### 13.1 AnimeTranscoder

`AnimeTranscoder` 已经从单纯 WPF 工具逐步演进出 Core / CLI / Services / Workflows 的形态。它适合承担确定性媒体执行：

- ffprobe 探测。
- ffmpeg 转码。
- 音频提取。
- 字幕烧录。
- 输出校验。
- 抽帧巡检。
- 批量任务。
- 日志和历史记录。

它不应该把所有 AI 推理直接塞进主程序。更好的边界是：主程序做 workflow host 和 media executor，AI 作为外部决策层，通过 JSON、SRT、TSV、manifest 交互。

### 13.2 GameSubtitleOCR

`GameSubtitleOCR` 当前是独立 Python OCR 子项目，使用 PaddleOCR / PP-OCRv5、FFmpeg、OpenCV。它支持：

- 采样若干视频帧检测字幕区域。
- OCR 参数调优。
- 按最佳参数流式抽取全视频字幕。
- 帧间去重、相似文本合并、时间戳生成。
- 输出 SRT。

它已经具备轻量工作流价值，但还不是完整的多帧融合系统，也不是 VLM QA 系统。

未来可以补：

- 同一句字幕的多帧候选保留。
- 字符级投票。
- 多 OCR profile 融合。
- 低置信截图导出。
- OCR 抽样 QA 报告。
- 与 VLM 或人工 review 的接口。

### 13.3 Ghost Yotei 工作流

Ghost Yotei 项目最重要的成果不是“做出了四个视频”，而是暴露了完整 AI 媒体流水线的真实边界。

它证明了：

- AI + 脚本 + OCR + ffmpeg 可以把流程跑通。
- 版本化、manifest、指针文件很重要。
- 英文主轴挂中文是正确方向。
- 局部高置信锚点比全局低置信覆盖更可靠。
- 抽帧检查硬字幕位置是必要步骤。

它也证明了：

- 覆盖率不等于质量。
- OCR 根基不稳会污染后续所有阶段。
- 低清源视频会严重限制上限。
- 模型不会自动替人建立质量门。
- 最终产物应该诚实标注为实验版或 beta。

## 14. 给未来 AI 的更强约束模板

以后不要只对 AI 说“继续做字幕”。应该给它更硬的边界。

### 14.1 停机条件模板

```text
你必须先执行质量评估，不能直接进入全量处理。

如果满足任一条件，停止并报告，不要继续生成最终字幕：
1. OCR 抽样明显错误率超过 10%。
2. 漏句或粘句超过 5%。
3. 中英匹配抽样中明显错配超过 10%。
4. 低置信行超过 20% 且没有人工或 VLM 复核计划。
5. 未完成硬字幕抽帧 QA。

停止时必须输出：
- 失败在哪一层。
- 证据文件。
- 建议的下一步。
- 是否需要更高清源。
```

### 14.2 只检查不修改模板

```text
本轮只允许读取和分析，不允许修改任何字幕或 JSON。

任务：
1. 读取指定 SRT/TSV/JSON。
2. 找出明显错配、空字幕、重复字幕、英文残留、专名错误。
3. 输出 markdown 报告。
4. 每个问题必须包含 part、时间码、英文、中文、问题类型和理由。

禁止：
- 不要补翻译。
- 不要删除行。
- 不要覆盖文件。
- 不要进入烧录阶段。
```

### 14.3 OCR 感知校对模板

```text
你现在是 OCR-aware 字幕校对员。

判断中文候选时必须区分：
- 可恢复 OCR 错字。
- 不可恢复残片。
- UI 文本混入。
- 句子错位。
- 模型生成译文。

不要因为 OCR 有一个错字就删除整句。
也不要为了填满而编造官方中文字幕。
```

### 14.4 时间轴截图 QA 模板

```text
你现在是字幕时间轴 QA。

我会给你截图或短片段。请判断：
1. 中文字幕是否遮挡原英文硬字幕。
2. 中文字幕是否和当前英文硬字幕语义对应。
3. 中文字幕是否像上一句或下一句。
4. 字号、位置、描边是否可读。
5. 是否存在 UI 文本被误当对白。

输出：
- layout: ok | overlap | unreadable
- timing: ok | early | late | uncertain
- semantic: ok | mismatch | uncertain
- note
```

## 15. 对你这次反思的直接回应

你的核心判断是对的：

> 工程问题的建构、设计、执行，每一步都值得重视。只要有一步没做好，后面会被放大。

但这里还要补一句：

> 真正的工程能力，不是永远不犯错，而是让错误尽早、尽小、尽可见。

这次最痛的地方不是 OCR 错了。OCR 错在这类项目里几乎必然发生。

真正的问题是：

- 没有在 OCR 阶段就量化错误率。
- 没有在样片阶段就判断上限。
- 没有在进入全量前设停机条件。
- 覆盖率指标给了错误的完成感。
- AI 继续执行了流程，但没有被制度要求停下来挑战目标。

所以未来要做的不是幻想 AI 自己“发脾气”，而是设计一个让 AI 必须说“不”的工作流。

例如：

- 没有高清源，AI 必须标记风险。
- OCR 抽样不达标，AI 必须停止全量。
- 低置信占比过高，AI 必须禁止标 final。
- 只完成结构覆盖，AI 必须说“这不是质量完成”。

这不是让 AI 变得更有性格，而是让工程流程更有刹车。

## 16. 最短结论

批处理速度，问的是这条链路能不能在全量素材上反复跑得动。

视觉语言模型，问的是系统能不能看画面判断字幕，而不只是处理文本。

多帧融合，问的是系统能不能利用同一句字幕在多个连续帧中重复出现这一事实，降低单帧 OCR 错误。

这三个能力共同决定 OCR 字幕工程的上限。

你这次真正学到的不是“AI 不可靠”，而是：

> AI 很强，但如果没有输入质量控制、阶段质量门、停机条件、版本追踪和诚实的产物分级，它会非常高效地把早期错误加工成一个看起来完整的最终文件。

下一次要升级的不是某一个提示词，而是整条工程制度。
