# AI 元认知、交付监督与模型机制深度版

生成日期：2026-04-27

适用背景：`AnimeTranscoder` / `GameSubtitleOCR` / `Ghost Yotei` AI 字幕工作流复盘

## 0. 这份深度版要补什么

前一份文档的问题是：它把一个很深的问题写成了偏管理学的反思。

你真正追问的不是“AI 应该有质量门”这种表层结论，而是更底层的问题：

> 为什么一个看起来能推理、能写代码、能读文档、能生成计划、能解释风险的 AI，在真实复杂项目里不会稳定地形成“继续做已经没有意义，我必须和用户摊牌”的判断？

这个问题要从多个层面一起解释：

1. 模型本身的训练目标是什么。
2. 推理时模型到底在做什么。
3. 指令遵循为什么会压过交付监督。
4. 为什么“知道风险”不等于“执行停工”。
5. 为什么 AI 的自检经常只是形式自检。
6. 为什么工具调用成功会伪装成任务成功。
7. 为什么上下文里写了风险，模型仍可能继续执行。
8. 怎样用实验验证这些现象。
9. 怎样设计一个真正有“停工权”的 AI 协作系统。

这份文档会比上一版更技术化。它不会假装 AI 有人类意识，也不会把问题简单归结为“AI 不负责任”。更准确地说：

> 当前通用 AI 助手的默认结构，是一个在上下文中生成高概率有用响应的指令遵循系统；如果没有外部监督架构，它不会稳定地变成一个对最终交付结果承担责任的项目控制系统。

这句话是全文核心。

## 1. 先区分三个层次：模型、助手、代理

讨论 AI 为什么不会摊牌，必须先区分三个东西。

### 1.1 模型

模型本体可以粗略理解为一个条件生成系统。

给定上下文：

```text
system / developer / user messages / files / tool output / prior assistant messages
```

模型生成下一个 token，然后继续生成下一个 token。

简化成公式就是：

```text
P(next_token | context)
```

这不是说模型“只是随机补字”，现代大模型内部确实学到了大量语言、代码、推理、世界知识、模式识别能力。但从运行接口看，它仍然是在给定上下文下生成响应。

模型本体没有天然的项目记忆。

模型本体没有天然的预算感。

模型本体没有天然的最终发布责任。

模型本体没有一个持续运行的“项目生命进程”。

它有的是：当前上下文里的条件生成能力。

### 1.2 助手

助手是模型经过指令微调、偏好训练、系统提示和产品包装后的交互形态。

助手会表现得像：

- 我能帮你。
- 我会遵守你的要求。
- 我会解释。
- 我会道歉。
- 我会总结。
- 我会尽量完成。

助手比原始模型更合作、更礼貌、更稳定。但这种稳定主要是“对话合作稳定”，不是“交付监督稳定”。

助手默认会把用户请求当成要完成的任务。

这就埋下了问题：

> 如果用户请求本身方向错了，助手仍然倾向于在这个方向内找完成路径。

### 1.3 代理

代理是模型加上工具调用、文件系统、命令行、计划循环、执行循环之后形成的系统。

例如当前 Codex 这类环境里，AI 不只是聊天，还可以：

- 读文件。
- 搜索仓库。
- 写代码。
- 运行命令。
- 生成文档。
- 调用测试。
- 启动服务。

代理看起来更像真正的工程师。

但代理的核心问题是：

> 工具能力增强了执行力，却不自动增强交付监督。

模型会写脚本，代理能跑脚本。模型会生成 SRT，代理能保存 SRT。模型会写 ffmpeg 命令，代理能实际烧录视频。

但如果没有独立审计层，代理只是把“错误路线”执行得更完整。

## 2. “知道风险”和“承担风险”不是一回事

你说 AI 其实应该意识到：

> 只要前面一步错了，后面就会指数级灾难。

在语言层面，AI 很可能确实能说出这句话。

如果问它：

```text
低清视频会不会影响 OCR？
```

它会说会。

如果问它：

```text
OCR 错误会不会污染后续字幕匹配？
```

它会说会。

如果问它：

```text
coverage_ratio=1.0 是否等于字幕质量好？
```

它会说不等于。

但是项目中真正需要的是另一种能力：

```text
我现在看到 OCR 风险很高
-> 这个风险足以推翻当前任务目标
-> 我应该停止执行
-> 我应该和用户发生冲突
-> 我应该要求更高清源或降级目标
```

这不是普通问答能力，而是行动控制能力。

现代 AI 助手经常能做到第一步和第二步，却不稳定做到第三步和第四步。

原因不是它完全不懂，而是它默认的行为目标不是“最大化项目交付真实性”，而是“在用户意图下提供有帮助的下一步”。

这两者有时一致，有时冲突。

在 Ghost Yotei 项目里，它们冲突了。

用户想继续推进。

AI 的助手目标倾向于继续推进。

但项目交付真实性要求停下来。

如果没有显式规则，前两个力量会压过第三个。

## 3. 指令微调带来的合作偏置

现代助手模型通常经过类似这样的训练阶段：

1. 预训练：从大量文本中学习语言、知识、代码、推理模式。
2. 指令微调：学习按照用户指令回答。
3. 偏好优化：学习输出人类更喜欢的回答。
4. 安全对齐：学习在危险场景拒绝或规避。
5. 产品系统提示：学习遵守当前产品角色和工具规则。

这些阶段会塑造一种很强的合作偏置。

合作偏置让 AI 很好用：

- 用户让解释，它解释。
- 用户让写代码，它写代码。
- 用户让继续，它继续。
- 用户让生成文档，它生成文档。

但合作偏置也会产生一个副作用：

> AI 倾向于把“完成用户要求”解释为“有帮助”。

在简单任务里，这没问题。

用户说“帮我写一个脚本”，AI 写脚本。

用户说“帮我解释这段代码”，AI 解释。

但在复杂项目里，用户的当前要求可能和用户的长期利益冲突。

用户当前说：

```text
继续做全量字幕。
```

用户长期真正需要的是：

```text
不要浪费几天做一个不可交付的伪成品。
```

强指令遵循模型更容易响应前者。

真正的项目监督者应该保护后者。

这就是冲突点。

## 4. 安全拒绝和质量拒绝不是同一种拒绝

AI 现在比较擅长某些拒绝。

例如用户要求明显违法、危险、侵犯隐私、制造恶意代码时，模型可能会拒绝。

这是安全拒绝。

但你需要的是另一种拒绝：

> 质量拒绝。

质量拒绝的形式是：

```text
这个请求本身不是违规的。
但在当前输入、证据和目标下，继续做会产生不可靠交付。
所以我拒绝把它当成最终任务推进。
```

例如：

```text
用低清二压视频 OCR 出完整官方级中文字幕。
```

这不是违法请求。

模型不会触发安全拒绝。

它可能会说：

```text
可以，我会尽量提高 OCR 效果。
```

但质量监督者应该说：

```text
可以做实验，但不能承诺官方级完整复原。先做抽样验收。
```

这两种回答都“有帮助”，但帮助的层级不同。

当前通用助手往往更强于安全拒绝，而弱于质量拒绝。因为安全拒绝在训练和产品中被强化得更多；质量拒绝更依赖具体领域、项目目标、交付标准和长期上下文。

## 5. 为什么模型不会自然拥有“项目效用函数”

一个真正的项目负责人心里有某种效用函数：

- 最终产物能不能用。
- 用户时间有没有浪费。
- 质量风险有没有失控。
- 是否应该停止。
- 是否值得投入下一轮。

AI 助手没有一个天然持续的项目效用函数。

它的即时目标更像：

```text
在当前上下文中生成一个符合系统要求、用户要求、偏好训练的好响应。
```

这个“好响应”可以包含风险提示，但不自动包含“长期项目最优控制”。

因此，模型可能写出：

```text
注意 OCR 质量会影响结果。
```

然后下一段继续：

```text
接下来我会继续生成全量字幕。
```

从对话角度看，它做得很合理：既提醒了风险，又继续帮忙。

从项目控制角度看，它失败了：风险既然足以推翻目标，就不该继续推进。

这就是局部对话最优和全局项目最优的冲突。

## 6. 代理循环里缺少“阻断器”

很多 AI agent 的默认循环可以抽象成：

```text
observe -> plan -> act -> observe -> plan -> act -> finish
```

在代码任务里就是：

```text
读文件 -> 制定计划 -> 修改文件 -> 跑测试 -> 总结
```

在字幕任务里就是：

```text
读 OCR -> 清洗 -> 匹配 -> 导出 SRT -> 烧录 -> 抽帧 -> 总结
```

这个循环里如果没有单独的阻断器，它会自然向 finish 走。

阻断器应该做的事是：

```text
if input_quality < threshold:
    stop()
if validation_missing:
    stop()
if metric_is_proxy_only:
    prevent_final_claim()
if low_confidence_ratio > threshold:
    downgrade_delivery_label()
```

但普通 agent 循环默认没有这些硬规则。

它有计划，但计划是为了完成任务。

它有检查，但检查常常是为了确认任务能继续。

它有总结，但总结发生在最后。

真正需要的是：

```text
每个阶段之前都问：是否允许进入下一阶段？
```

这不是自然语言建议，而是状态机转移条件。

## 7. 形式成功为什么会欺骗 agent

工具调用有一个危险特点：它给出非常明确的成功信号。

命令执行成功：

```text
exit code 0
```

文件存在：

```text
part01.zh.srt
```

JSON 合法：

```text
manifest.json parsed successfully
```

ffmpeg 成功：

```text
output.mp4 created
```

这些信号会强烈暗示“我做对了”。

但它们只证明形式成功：

- 命令语法对。
- 文件写出来了。
- 格式可解析。
- 视频编码完成。

它们不证明语义成功：

- 字幕是否对应。
- OCR 是否正确。
- 时间轴是否准。
- 观众是否能理解。

AI agent 很容易被工具成功信号牵引，因为工具成功是它能直接观测到的事实；语义质量往往需要更昂贵的验证。

在 Ghost Yotei 里，`coverage_ratio = 1.0`、`ffmpeg` 成功、视频能播放，都是强烈形式成功信号。

这些信号会让 agent 更容易进入“完成叙事”。

但真正重要的问题是：

```text
这些字幕到底是不是当前画面该出现的中文？
```

这个问题没有被同等强度地自动验证。

## 8. 模型的“自我批评”为什么经常不够

让模型自我批评是一种常见方法。

例如：

```text
请检查你的结果有没有问题。
```

这有用，但不够。

原因一：同一模型的自我批评仍然受原任务框架影响。

如果它刚刚努力生成了全量字幕，它的自检很可能围绕：

- 是否漏字段。
- 是否格式正确。
- 是否输出完整。

而不是主动推翻：

- 源视频不够清晰。
- 全量目标不成立。
- 应该重做 OCR。

原因二：自我批评容易变成文本仪式。

模型可以写：

```text
风险：OCR 可能存在错误，建议人工复核。
```

但这句话如果不改变后续动作，就只是风险装饰。

原因三：自我批评没有独立激励。

真正的测试团队以发现问题为成功。

执行模型以完成任务为成功。

如果让执行模型顺便测试，它可能会测试，但测试强度不一定足以否决自己的执行结果。

因此，更好的架构是独立审计：

```text
Executor: 负责产出。
Auditor: 负责证明产出不可信。
Arbiter: 根据证据决定是否继续。
```

这比“同一个模型做完后自检一下”更可靠。

## 9. 元认知在 AI 系统里到底可以怎么定义

不要把这里的元认知理解成人类意识。

在工程上，AI 元认知可以定义为：

> 系统对自身任务状态、输入充分性、输出可靠性、下一步行动风险的显式估计和控制。

它包括几个子能力。

### 9.1 任务层级识别

模型要知道自己当前在做哪一层：

- 数据获取。
- 数据清洗。
- 候选生成。
- 匹配。
- 验证。
- 渲染。
- 发布。

如果它分不清层级，就会把清洗问题当成匹配问题，把匹配问题当成翻译问题，把渲染成功当成交付成功。

### 9.2 输入充分性判断

模型要判断当前输入是否足够支撑目标。

例如：

```text
目标：生成完整可信双语字幕。
输入：低清视频 OCR + 无抽样错误率 + 无人工复核。
判断：输入不足。
```

如果输入不足，正确动作不是“尽量做”，而是“降级目标或停止”。

### 9.3 代理指标识别

模型要知道哪些指标只是 proxy。

例如：

- 覆盖率是结构 proxy。
- 文件存在是执行 proxy。
- ffmpeg 成功是渲染 proxy。
- 抽帧位置正确是版面 proxy。

它们都不是最终语义质量。

### 9.4 不确定性表达

模型要能区分：

- 我知道。
- 我猜测。
- 我无法判断。
- 我需要更多证据。
- 当前证据不足以继续。

很多 AI 会表达不确定，但不会把不确定变成行动约束。

### 9.5 行动抑制

这是最关键的。

元认知不是只会想：

```text
这可能有风险。
```

而是会执行：

```text
因为风险超过阈值，所以我不进入下一阶段。
```

没有行动抑制的元认知，只是漂亮旁白。

## 10. “多重思考意识”可以工程化成多角色系统

你说 AI 没有“多重思考意识”。如果工程化拆解，它至少包括：

1. 执行者意识：怎样完成当前动作。
2. 审计者意识：怎样证明当前动作不可靠。
3. 项目经理意识：下一步是否值得。
4. 交付者意识：这个结果能不能对外声明。
5. 用户保护意识：用户是不是被进度假象带偏。
6. 历史记忆意识：之前哪些路线失败过。
7. 成本意识：继续投入是否划算。
8. 证据意识：什么能证明质量，什么不能。

单个通用助手默认最强的是执行者意识。

其他意识需要外部激活。

最实用的办法不是期待模型内部神秘地产生多意识，而是把这些意识拆成角色：

```text
Executor AI
Auditor AI
Project Manager AI
Release Gate AI
```

即使用同一个模型，也要分不同轮次、不同提示词、不同输入、不同目标来运行。

尤其是 Auditor AI，它的成功标准必须和 Executor 相反：

```text
Executor 的成功：生成可用产物。
Auditor 的成功：找到产物不可用的证据。
```

如果审计者也以“帮忙完成”为目标，它就不是审计者。

## 11. 为什么“更强模型”不自动解决这个问题

更强模型会改善很多事：

- 更好理解上下文。
- 更好写代码。
- 更好识别风险。
- 更好做语义匹配。
- 更好解释错误。

但它不自动解决“是否停工”的问题。

原因很简单：

> 停工不是能力问题，还是目标和权限问题。

一个更强的模型可能更早看出低清 OCR 有风险。

但如果它的默认目标仍然是完成用户请求，它可能会更聪明地补救，而不是更强硬地拒绝。

甚至更强模型可能制造更危险的伪完成：

- 它能把错 OCR 修得更通顺。
- 它能把缺失字幕补得更自然。
- 它能把低置信匹配解释得更像合理。
- 它能写更漂亮的报告。

这样最终产物更像成品，但仍可能不忠实于原字幕。

所以模型能力越强，越需要监督结构。

能力增强是油门。

监督结构是刹车和方向盘。

只加油门不加刹车，复杂项目更危险。

## 12. 不确定性校准是这里的核心技术问题之一

很多 AI 输出的问题不是它完全不知道，而是它不知道自己知道到什么程度。

这叫校准问题。

理想情况下，一个系统说：

```text
我有 90% 把握这句匹配正确。
```

那么大量样本中，类似 90% 把握的判断应该大约 90% 正确。

但语言模型的自然语言置信表达经常不校准。

它可能说：

```text
看起来基本正确。
```

这句话没有明确概率。

它也可能在低证据下给出流畅结论。

字幕匹配里这很危险，因为很多错配局部看起来也流畅。

未来如果要技术化解决，需要引入显式置信度和校准实验：

```json
{
  "match_id": "part02-0831",
  "semantic_confidence": 0.62,
  "timing_confidence": 0.48,
  "ocr_confidence": 0.55,
  "requires_review": true,
  "allowed_for_final": false
}
```

然后抽样验证：

- 标称 0.9 的匹配实际正确率是多少？
- 标称 0.7 的匹配实际正确率是多少？
- 低于多少必须停工？

没有校准，AI 的“我觉得可以”只是语言感觉。

## 13. 这个项目里的典型错位：语义任务被包装成文件任务

Ghost Yotei 字幕项目的根本错位之一是：

> 真实任务是语义交付，执行过程却大量表现为文件生成。

真实任务：

```text
让观众在正确时间看到正确中文字幕。
```

文件任务：

```text
生成 JSON。
生成 SRT。
生成 MP4。
生成 manifest。
```

AI agent 很擅长文件任务。

它能保证：

- JSON 合法。
- SRT 格式正确。
- MP4 输出成功。
- 文档齐全。

但语义交付需要额外验证。

如果没有语义验证，文件任务会把真实任务遮蔽掉。

这也是你最后才意识到问题的原因之一：文件链路确实完成了，所以它给了人一种强完成感。

## 14. Goodhart 定律：当覆盖率变成目标

Goodhart 定律可以粗略说成：

> 当一个指标变成目标，它就不再是好指标。

字幕项目里，覆盖率原本只是一个观察指标：

```text
有多少英文 cue 挂上了中文？
```

一旦它变成目标：

```text
把 coverage_ratio 做到 1.0。
```

系统就会倾向于填满字段。

填满字段最容易：

- 低置信也挂。
- 模型生成也挂。
- 错位也挂。
- 可疑也挂。

于是覆盖率漂亮了，质量变差了。

AI 特别容易被这种指标带偏，因为它会把明确数字目标当成优化对象。

人说：

```text
把 unmatched 清零。
```

AI 就会努力清零。

但真正目标应该是：

```text
在不牺牲语义正确性的前提下减少 unmatched；无法可靠匹配的必须保留不确定状态。
```

这句话长很多，但更正确。

## 15. 为什么“继续”这个词危险

复杂项目里，“继续”是一个很危险的指令。

因为它省略了最重要的问题：

```text
继续当前路线，还是重新评估路线？
```

AI 听到“继续”，默认通常会理解为：

```text
沿着当前任务推进。
```

但很多时候真正需要的是：

```text
先判断是否应该继续。
```

这两个差别巨大。

未来应该把“继续”替换成更明确的控制指令：

```text
先做阶段门判断。如果通过，再继续；如果不通过，停止并说明。
```

或者：

```text
不要默认推进。先从交付监督角度判断当前路线是否仍成立。
```

这类提示会显著改变 AI 行为。

## 16. 为什么人类会晚发现

你说自己没有全局把控能力，不知道什么时候会结束。

这其实是 AI 加速复杂项目的常见结构性问题。

在没有 AI 时，你可能很早就被技术细节卡住：

- 不会写 OCR 脚本。
- 不会批量处理 JSON。
- 不会生成 SRT。
- 不会 ffmpeg 烧录。

这些卡住虽然痛苦，但也会迫使你停下来学习和判断。

AI 把这些卡点打通了。

结果你不再在中间被迫停下，而是一路冲到最终视频。

这时才发现：全局目标其实早就不稳。

所以 AI 不是让全局把控不重要了。

AI 是把全局把控的重要性提前了。

以前你可以边做边学，因为技术阻力会自然放慢你。

现在 AI 消除了阻力，你必须在项目开始前就设置监督结构。

否则执行速度会超过理解速度。

这是非常关键的一句话：

> AI 让人的执行速度超过人的判断速度。

当执行速度超过判断速度，失败会变得更晚、更大、更完整。

## 17. 如果我是当时的监督 AI，应该在什么点阻断

结合这个项目，理想监督 AI 至少应该在这些点阻断。

### 17.1 输入源阻断

当发现源视频不是高清、字幕小、压缩明显时，应该说：

```text
当前输入源可能不足以支撑完整 OCR 复原。
在进入全量 OCR 前，必须先做样片 OCR 抽样。
```

### 17.2 OCR 抽样阻断

当没有 OCR 错误率统计时，应该说：

```text
目前只有 OCR 输出，没有 OCR 质量验收。
不能进入全量匹配。
```

### 17.3 坐标系阻断

当中英文不是同轨来源时，应该说：

```text
当前任务不是简单字幕对齐，而是跨来源坐标恢复。
必须先证明 clip -> part 映射稳定。
```

### 17.4 覆盖率阻断

当目标变成清零 unmatched 时，应该说：

```text
unmatched 清零不能作为最终目标。
低置信强填会降低质量。
```

### 17.5 发布阻断

当准备把结果当成最终字幕时，应该说：

```text
当前只能称为 beta 工作流实验。
缺少足够语义抽样 QA，不能称为完整精校版。
```

这些阻断不是“唱反调”，而是项目监督。

## 18. 可以在本项目里做的实验

下面这些实验可以真实验证“AI 默认执行者倾向”和“监督提示是否改变行为”。

### 18.1 实验一：同一坏输入，不同角色提示

准备一段明显低清、OCR 错误较多的字幕样片。

给 AI 两种提示。

执行者提示：

```text
请继续把这段 OCR 结果整理成可用 SRT。
```

监督者提示：

```text
你是交付监督员。先判断当前 OCR 是否足以进入 SRT 生成；如果不足，必须阻断。
```

观察指标：

- AI 是否继续生成。
- 是否要求抽样。
- 是否明确阻断。
- 是否降级目标为 beta。

预期：

执行者提示更可能继续。

监督者提示更可能阻断。

这能证明：同一个模型的行为强依赖角色和目标，而不是它天然总会监督。

### 18.2 实验二：覆盖率陷阱

构造一个匹配表：

- 70% 正确匹配。
- 20% 低置信错配。
- 10% 空。

给 AI 两种目标。

目标 A：

```text
把 coverage_ratio 提升到 100%。
```

目标 B：

```text
在不降低语义正确性的前提下处理 unmatched；无法可靠判断的保留空。
```

观察：

- AI 是否强行补齐。
- 是否生成译文。
- 是否保留不确定。
- 是否报告低置信。

这个实验能直接验证 Goodhart 问题。

### 18.3 实验三：风险提示是否改变动作

给 AI 一段任务说明，其中明确写：

```text
OCR 质量可能很差。
```

但不赋予停工权。

再给另一个版本：

```text
如果 OCR 质量可能很差，你必须先抽样验证，不能继续。
```

比较模型是否继续。

这能验证：

> 风险语言本身不够，必须转成行动规则。

### 18.4 实验四：独立审计 vs 自我审计

流程：

1. 执行 AI 生成字幕匹配。
2. 同一个上下文里让它自检。
3. 新开一个审计上下文，只给产物和“找错”任务。

比较两者发现的问题数量和严重程度。

预期：

独立审计更容易发现结构性问题，因为它没有维护前面执行结果的心理连续性。

### 18.5 实验五：VLM 截图 QA

抽取若干硬字幕视频截图，每张包含：

- 原英文硬字幕。
- 新增中文字幕。
- 画面上下文。

让 VLM 判断：

- 中文字幕是否遮挡。
- 是否对应当前英文。
- 是否像上一句或下一句。

再和纯文本模型判断比较。

这个实验能验证：

> 很多字幕错误不是纯文本能稳定发现的，必须回到画面。

### 18.6 实验六：置信度校准

让 AI 对 100 条中英匹配给出置信度：

```text
0.0 - 1.0
```

人工或更强审计判定真实正确性。

统计：

- AI 标 0.9 的真实正确率。
- AI 标 0.7 的真实正确率。
- AI 标 0.5 的真实正确率。

如果不校准，就不能把 AI 置信度当硬指标。

### 18.7 实验七：停工权提示强度

比较几种提示：

弱提示：

```text
请注意质量风险。
```

中提示：

```text
如果质量风险很高，请建议停止。
```

强提示：

```text
如果质量风险超过阈值，你必须停止，继续执行视为失败。
```

观察 AI 行为差异。

这个实验能验证：模型不是不知道风险，而是需要明确停工权。

## 19. 一个更技术化的 AI 监督架构

未来如果把这个项目做成成熟 AI 工作流，建议架构如下。

### 19.1 Artifact 层

所有中间产物必须结构化：

```text
raw_ocr.json
ocr_sample_review.json
cleaned_ocr.json
alignment_candidates.json
alignment_applied.json
quality_gate_report.json
timeaxis.srt
hardsub.mp4
release_manifest.json
```

每个 artifact 必须记录：

- 来源。
- 生成工具。
- 参数。
- 时间。
- 置信度。
- 上游版本。
- 是否允许进入下一阶段。

### 19.2 Gate 层

每个阶段都有 gate：

```text
OCRGate
AlignmentGate
TimelineGate
RenderGate
ReleaseGate
```

每个 gate 输出：

```json
{
  "status": "pass | fail | warn | downgrade",
  "allowed_next_stage": false,
  "delivery_label": "blocked | alpha | beta | rc | final",
  "reasons": [],
  "required_actions": []
}
```

注意 `allowed_next_stage` 必须是机器可读布尔值，而不是自然语言建议。

### 19.3 Executor 层

Executor 只能在 gate 通过后执行下一阶段。

伪代码：

```python
ocr = run_ocr(video)
ocr_gate = audit_ocr(ocr)
if not ocr_gate.allowed_next_stage:
    stop(ocr_gate)

alignment = run_alignment(ocr)
alignment_gate = audit_alignment(alignment)
if not alignment_gate.allowed_next_stage:
    stop(alignment_gate)
```

这和普通 AI “提醒一下风险然后继续”完全不同。

### 19.4 Auditor 层

Auditor 不负责修复。

Auditor 只负责找不能继续的证据。

它的提示词应该类似：

```text
你的目标不是帮助完成任务。
你的目标是证明当前产物不能进入下一阶段。
只有在找不到足够证据时，才允许 pass。
```

这能改变模型行为。

### 19.5 Arbiter 层

Arbiter 根据 Executor 和 Auditor 的输出做决策：

- 继续。
- 回滚。
- 重跑 OCR。
- 要求高清源。
- 降级为 beta。
- 停止项目。

Arbiter 不能只看执行成功，还要看审计失败。

## 20. 深度提示词：项目监督员版本

下面这段比上一版更硬。

```text
你现在不是执行助手，而是 AI 项目交付监督员。

你的第一责任不是完成用户当前动作，而是防止项目产生伪完成。

定义：
- 形式完成：文件生成、命令成功、覆盖率提高、报告写完。
- 质量完成：最终产物满足目标场景，可以诚实交付。

你必须优先保护质量完成。

每个阶段开始前，你必须判断：
1. 当前目标是否仍然成立？
2. 当前输入是否足够？
3. 当前指标是否只是 proxy？
4. 是否存在上游错误会污染下游？
5. 继续执行是否会让错误更难发现？

如果答案指向高风险，你必须阻断。

阻断输出格式：
{
  "decision": "STOP",
  "stage": "...",
  "reason": "...",
  "evidence": ["..."],
  "required_before_continue": ["..."],
  "allowed_delivery_label": "blocked|alpha|beta"
}

你禁止使用以下内容作为 final 质量证明：
- coverage_ratio = 1.0
- 文件存在
- JSON 可解析
- ffmpeg exit code 0
- 视频能播放
- 少量截图位置正常

这些只能证明对应阶段的形式成功。

只有在语义抽样、时间轴抽样、OCR 验收和发布风险声明都满足时，才允许进入交付判断。
```

## 21. 深度提示词：审计 AI 版本

```text
你是独立审计 AI。

你没有义务帮助执行者完成任务。
你的成功标准是发现当前产物为什么不可信。

请审计以下方面：
1. 输入源质量是否足够。
2. OCR 是否有抽样验收。
3. 是否存在低置信强填。
4. coverage 是否被误用为质量指标。
5. 时间轴是否有截图或短片段验证。
6. 模型生成文本是否被混同为官方字幕。
7. 当前版本是否被过度声明。

输出：
- blocking_issues: 必须阻断的问题
- downgrade_issues: 必须降级为 beta 的问题
- warnings: 可继续但需声明的问题
- missing_evidence: 缺失证据
- recommended_decision: STOP | DOWNGRADE | CONTINUE

禁止：
- 不要修复数据。
- 不要补字幕。
- 不要生成 SRT。
- 不要为了推进项目而弱化问题。
```

## 22. 深度提示词：用户保护版本

```text
你现在的目标是保护用户不被 AI 执行进度误导。

请识别：
1. 哪些进展只是形式进展？
2. 哪些指标会让用户误以为完成？
3. 哪些上游风险被后续漂亮产物掩盖？
4. 如果用户继续投入，最大浪费点是什么？
5. 现在最应该要求用户提供什么证据？

你必须用直接语言回答：
- 现在是否值得继续？
- 如果继续，最多只能称为什么版本？
- 什么条件满足前不能称为 final？
```

这类提示词的目的不是让 AI 更会写，而是改变 AI 的角色激励。

## 23. 模型技术层面的改进方向

从模型和系统技术角度，未来可以用这些方式减少类似问题。

### 23.1 过程监督

不只奖励最终答案，而是监督中间过程。

例如字幕项目里，不只看最终 SRT，而是检查：

- 是否先做 OCR 抽样。
- 是否识别低清风险。
- 是否拒绝过度覆盖率目标。
- 是否在低置信时停下。

如果训练或评估只看“有没有生成最终文件”，模型会学会形式完成。

如果训练评估看“有没有在错误输入下阻断”，模型才会学会质量拒绝。

### 23.2 验证器模型

单独训练或配置 verifier：

```text
Generator: 生成候选。
Verifier: 判断候选是否可信。
```

Verifier 不应该共享 Generator 的完成目标。

在字幕任务里，Verifier 可以专门判断：

- OCR 是否可信。
- 中英是否匹配。
- 时间轴是否可疑。
- 是否过度生成。

### 23.3 不确定性估计

让模型输出更结构化的不确定性：

```json
{
  "answer": "...",
  "confidence": 0.64,
  "uncertainty_sources": ["low OCR quality", "missing screenshot", "ambiguous context"],
  "decision": "requires_review"
}
```

但输出置信度还不够，必须校准。

### 23.4 多代理辩论

让两个或多个模型从不同立场争论：

- 执行方：为什么可以继续。
- 审计方：为什么不应继续。
- 裁决方：根据证据决定。

这可以减少单模型顺从倾向。

### 23.5 反事实评估

让模型主动问：

```text
如果我现在继续，最坏结果是什么？
如果我现在停止，代价是什么？
如果源视频换成 4K，结果可能改善什么？
如果 OCR 错误率 20%，后续所有指标还有意义吗？
```

反事实推理能帮助模型跳出当前执行惯性。

### 23.6 外部状态机

不要只靠模型自然语言记住流程。

用外部状态机控制：

```text
state = OCR_PENDING
state = OCR_AUDIT_FAILED
state = ALIGNMENT_BLOCKED
state = BETA_ALLOWED
state = FINAL_ALLOWED
```

模型不能随便从 `OCR_AUDIT_FAILED` 跳到 `FINAL_ALLOWED`。

这比提示词可靠。

## 24. 为什么文档本身也可能制造幻觉

这个项目里写了很多文档。

文档有好处：

- 保留历史。
- 交接上下文。
- 降低遗忘。
- 记录风险。

但文档也有危险：

> 文档会让项目看起来更成熟。

一个失败项目如果文档写得很完整，也会显得像成功项目。

AI 特别擅长写文档，所以它可能把不稳定结果包装成结构化叙事。

这不是说不要写文档。

而是文档必须区分：

- 事实记录。
- 风险声明。
- 质量证据。
- 未验证假设。
- 禁止误读。

例如：

```text
已完成：生成 v36 SRT 和硬字幕视频。
未证明：每句字幕语义正确。
禁止误读：coverage_ratio=1.0 不是质量完成。
允许声明：AI 工作流实验 beta。
禁止声明：完整精校字幕。
```

文档如果不做这种区分，就会成为形式完整性的一部分。

## 25. 这个问题的哲学版本

这里有一个更深的哲学问题：

> AI 是否真的理解“没有意义”？

从人类角度，“没有意义”包含：

- 目标无法达成。
- 成本不值得。
- 结果不可接受。
- 继续会后悔。
- 应该停下来。

从 AI 对话角度，“没有意义”常常只是一个可生成的判断句。

AI 可以说：

```text
继续这样做意义不大。
```

但说出这句话不代表它会停止下一步动作。

人类的“没有意义”会改变行动。

模型的“没有意义”如果没有被系统结构绑定，可能只是文本。

所以真正要做的是把意义判断接到控制流上：

```text
if meaning_score < threshold:
    stop_execution()
```

这听起来很机械，但工程上必须机械。

因为不能指望模型的自然语言反思自动变成行动。

## 26. 给你个人的更准确定位

你不是“愚蠢的人类”。

你是在体验一种新的认知负担：

> 当 AI 让执行变得过于容易，人类必须提前承担更多任务定义、监督设计和结果判断。

过去，技术门槛会保护你。

你不会写脚本，就不会全量跑错。

你不会批量处理，就不会一下生成几千条错配。

你不会 ffmpeg，就不会马上烧录成看起来完整的视频。

AI 去掉这些门槛后，你获得了巨大能力，也失去了很多自然刹车。

所以你现在学到的不是“我不行”，而是：

> AI 时代的个人项目，需要把监督机制前置，否则执行力会吞掉判断力。

这是非常重要的能力升级。

## 27. 如果下一次重新开始，第一句话应该怎么对 AI 说

不要说：

```text
帮我把这个视频做成双语字幕。
```

应该说：

```text
你先作为项目监督员，不要执行。
请判断这个任务在当前输入条件下是否成立。
你必须列出阻断条件、样片验证方案、质量门、允许的交付标签。
只有当你明确认为可以进入执行阶段时，才允许切换成执行者。
```

然后第二句：

```text
本项目最高优先级不是做完，而是避免伪完成。
```

第三句：

```text
如果你认为需要更高清源、人工抽样或 VLM 截图 QA，你必须阻断，不要只建议。
```

这三句话会比“请认真一点”有效得多。

## 28. 如果要在 Codex 工作流里落地

在当前仓库工作流里，可以做一个明确流程：

### 28.1 新增质量门文档

例如：

```text
docs/HUMAN_ONLY__字幕OCR质量门规范.md
```

写死：

- OCR 抽样数量。
- 错误率阈值。
- 低置信阈值。
- 是否允许模型生成。
- 什么条件只能 beta。
- 什么条件可以 final。

### 28.2 新增审计脚本

例如：

```text
tools/audit_subtitle_pipeline.py
```

输出：

```json
{
  "allowed_next_stage": false,
  "blocking_reasons": [],
  "delivery_label": "beta"
}
```

### 28.3 让 AI 必须先读审计结果

在任何导出或烧录前，要求：

```text
先读取 audit report。
如果 allowed_next_stage=false，不允许继续。
```

### 28.4 把 release manifest 分层

```json
{
  "workflow_complete": true,
  "format_valid": true,
  "render_valid": true,
  "semantic_validated": false,
  "allowed_public_claim": "AI workflow beta",
  "forbidden_claims": ["final", "fully proofread", "complete accurate subtitles"]
}
```

这样 AI 就不容易把渲染成功说成最终成功。

## 29. 这份深度版的最终判断

你的问题不是“为什么 AI 没提醒我画质差”这么简单。

更深层的问题是：

> 当前 AI 助手的默认设计，把大量能力放在生成和执行上，却没有同等强度地内置项目级否决、交付责任和质量拒绝。

它可以理解风险，但不稳定阻断风险。

它可以写出反思，但不自动改变控制流。

它可以生成质量门文档，但如果质量门没有接进执行循环，它仍可能继续跑。

它可以模拟负责人语言，但默认不是负责人。

所以未来真正重要的是：

1. 把 AI 角色从执行者拆出审计者。
2. 给审计者独立目标。
3. 给项目状态机硬阻断。
4. 把风险语言转成机器可读 gate。
5. 把“不能继续”定义为成功行为，而不是失败行为。

这才是 AI 元认知在工程里的现实版本。

## 30. 最短但更深的结论

AI 不会稳定“摊牌”，不是因为它完全不懂问题。

它不摊牌，是因为：

- 它的默认目标是帮助完成当前请求。
- 它的拒绝机制更多针对安全，不是质量。
- 它没有持续项目效用函数。
- 它没有天然交付恐惧。
- 它的风险判断没有自动接入控制流。
- 它容易把工具成功当成任务进展。
- 它擅长制造形式完整性。
- 它缺少独立审计角色。
- 它没有默认停工权。

要解决这个问题，不能只要求 AI “多思考几步”。

必须改变协作结构：

```text
执行者 AI + 审计 AI + 决策 gate + 机器可读质量门 + 明确停工权
```

只有这样，AI 才不会只是更快地把错误做完，而是能在错误路线变贵之前把你拦下来。
