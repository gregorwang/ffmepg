## 0. 先说结论

这批 Bilibili 封面并不是用“原生多模态模型直接生成图片”的方式做出来的。

更准确地说，它是一个传统图像工程流程：

1. 从视频里抽取游戏实机画面。
2. 人工或半自动挑选构图比较适合做封面的帧。
3. 以这些帧为底图。
4. 用程序化绘图方式叠加标题、分集标签、说明文字、角标、色条、遮罩、底部风险声明。
5. 导出 `1920x1200` 的 PNG。
6. 再做一张 contact sheet 方便横向比较。

也就是说，封面的“画面内容”来自真实视频帧，封面的“文字排版”来自程序化的 2D 绘制。

它看起来像用了 OCR，是因为底图本身保留了视频原画里的英文字幕、水印、黑边等信息；同时整个项目上下文确实有 OCR / 字幕对齐 / 字幕校对工作流，所以封面文案也围绕“中英字幕”“官方英文硬字幕”“官方中文字幕”“时间轴匹配”这些 OCR / 字幕工程概念展开。

但在封面生成这一步，核心能力不是 OCR。

真正起作用的是：

- 对视频帧内容的选择。
- 对 Bilibili 封面尺寸和可读性的理解。
- 对文字层级、字体粗细、留白、遮罩、色彩和对比度的程序化控制。
- 对目标用户一眼能看懂什么信息的判断。

换句话说：

> 这不是“AI 画了一张封面”，而是“AI 挑了一张合适的游戏截图，然后像设计师一样用代码排版了一张封面”。

## 1. 当前目录里能确认的事实

从文件尺寸和目录结构可以确认：

### 1.1 源帧是 1920x1080

`source_frames` 下的四张图都是 `1920x1080`：

- `part01.png`: `1920x1080`
- `part02.png`: `1920x1080`
- `part03.png`: `1920x1080`
- `part04.png`: `1920x1080`

这说明封面底图不是凭空生成的，而是来自原始视频抽帧。

### 1.2 最终封面是 1920x1200

最终导出的五张封面都是 `1920x1200`：

- `ghost_yotei_part01_cover.png`: `1920x1200`
- `ghost_yotei_part02_cover.png`: `1920x1200`
- `ghost_yotei_part03_cover.png`: `1920x1200`
- `ghost_yotei_part04_cover.png`: `1920x1200`
- `ghost_yotei_workflow_beta_cover.png`: `1920x1200`

这点很重要。

原始视频帧是 `16:9`，最终封面是 `8:5`。封面没有简单裁成 16:9，而是把纵向空间加高，让顶部标题区和底部风险声明区有更多呼吸空间。

Bilibili 视频封面常见显示环境里，缩略图会被各种 UI 裁切、压缩、覆盖角标。用 `1920x1200` 这种略高的画布，比纯 `1920x1080` 更容易给标题和底部说明留空间。

### 1.3 `part02_candidates` 是候选帧筛选目录

`part02_candidates` 下有：

- `part02_0500.png`
- `part02_1200.png`
- `part02_2500.png`
- `part02_4500.png`
- `contact_sheet.png`

这说明至少对 `part02` 做过候选帧横向比较。

文件名里的 `0500`、`1200`、`2500`、`4500` 很像抽帧时间点，常见含义是：

- 500 秒附近
- 1200 秒附近
- 2500 秒附近
- 4500 秒附近

也可能是脚本内部用于标记候选序号或秒数的简写。但结合封面候选语境，更可能是“从长视频不同时间点抽出若干帧，然后比较哪张适合做封面”。

### 1.4 `cover_contact_sheet.png` 是成品总览

`cover_contact_sheet.png` 是 `1200x1600`，把五张成品封面排成一个对比板：

- part01
- part02
- part03
- part04
- workflow beta

这不是用户最终上传的封面，而是 review 用图。

contact sheet 的价值是：

- 同时看 5 张封面的风格是否统一。
- 检查标题大小是否一致。
- 检查每个 part 的色彩区分是否清楚。
- 检查底部风险声明是否可读。
- 检查封面在缩小后是否仍然能被识别。

## 2. 为什么看起来像用了 OCR

你说“看起来用了 OCR 相关能力”，这个判断很自然，因为图里有几个强烈信号。

### 2.1 底图里保留了原视频字幕

例如 `source_frames/part02.png` 底部有英文字幕：

`Yone: Why not play a game with your mother then?`

最终封面 `ghost_yotei_part02_cover.png` 也保留了这个字幕。

这会让人感觉系统“读懂了字幕”，但其实封面生成不一定需要 OCR 识别这行字。

封面可以直接把原视频帧作为底图使用，字幕作为画面的一部分自然留下来。只要不把底部裁掉，字幕就会继续存在。

### 2.2 封面文案围绕字幕工程

封面上的文字包括：

- `中英字幕`
- `英语学习自用`
- `官方英文硬字幕`
- `官方中文字幕`
- `非完整精校`
- `字幕时间轴可能存在误差`
- `workflow proof / beta release`
- `翻译校对 + 时间轴匹配`
- `OCR 清洗与字幕复刻`

这些词来自项目真实工作流：

- 英文硬字幕 OCR。
- 中文字幕 OCR。
- 中英对齐。
- 时间轴匹配。
- LLM 辅助校对。
- beta 交付声明。

所以看起来像 OCR 参与了封面生成。

但这更准确地说是“封面文案总结了 OCR 工作流”，不是“封面视觉由 OCR 生成”。

### 2.3 真实流程里 OCR 是上游语境，不是封面绘制引擎

这批封面和 OCR 的关系可以这样拆：

| 环节 | 是否可能使用 OCR | 作用 |
| --- | --- | --- |
| 视频字幕提取 | 是 | 识别英文/中文硬字幕 |
| 字幕对齐与校对 | 是 | 生成或修正双语字幕 |
| 封面文案确定 | 间接受 OCR 工作流影响 | 写出“官方英文硬字幕”“中文字幕”等标签 |
| 封面图像绘制 | 否，不需要 OCR | 用 Pillow / 图像绘制 API 叠加文字和形状 |
| 成品质量判断 | 可用视觉检查，不一定 OCR | 看缩略图是否可读 |

这也是它能做得比较稳定的原因。

让 OCR 去“生成封面”是不合适的；OCR 擅长读字，不擅长排版。封面排版应该交给确定性的绘图逻辑。

## 3. 这批封面的实际生成思路

可以把流程理解为：

```text
长视频
  -> 按 part 抽帧
  -> 挑选有代表性的源帧
  -> 统一画布到 1920x1200
  -> 放入源帧
  -> 添加顶部暗色遮罩
  -> 添加右上角斜切色块
  -> 添加左侧主标题卡片
  -> 添加分集色条
  -> 添加功能标签
  -> 添加底部 beta 声明
  -> 导出单张封面
  -> 生成 contact sheet
```

这个流程里的每一步都可以用代码完全复现。

最可能使用的技术栈是 Python + Pillow。

原因：

1. 目录里的产物都是 PNG。
2. 字体、圆角矩形、半透明遮罩、斜切色块、描边标签都很像 Pillow 的 `ImageDraw` 绘制结果。
3. 没有看到 Photoshop / Figma 工程文件。
4. 也没有看到 Stable Diffusion、DALL-E、ComfyUI 等生成式图片工作流的痕迹。
5. 文字位置高度一致，说明更像是坐标驱动，而不是手工拖拽。

## 4. 抽帧阶段

### 4.1 抽帧目标

封面不是随便截一张图。

用于封面的帧需要满足几个条件：

1. 主体清楚。
2. 背景有辨识度。
3. 左上区域或标题区域有足够干净的空间。
4. 画面明暗对比足够。
5. 不要有太多 UI 干扰。
6. 底部字幕不应该遮挡封面主要信息。
7. 缩小到 Bilibili 信息流尺寸后仍然能看出主题。

`source_frames/part02.png` 是一个典型例子：

- 右侧是房屋和人物，有剧情感。
- 左侧和中上方是天空、湖面、远山，视觉上比较干净。
- 左上区域适合放大标题。
- 画面颜色明亮，缩略图可读性强。
- 底部英文字幕保留后还能强化“英文学习 / 字幕工程”的语境。

### 4.2 可能的抽帧命令

如果从视频抽取静态帧，通常会用 ffmpeg：

```powershell
ffmpeg -ss 00:08:20 -i part02.mp4 -frames:v 1 -q:v 2 scratch\bilibili_covers_v1\part02_candidates\part02_0500.png
ffmpeg -ss 00:20:00 -i part02.mp4 -frames:v 1 -q:v 2 scratch\bilibili_covers_v1\part02_candidates\part02_1200.png
ffmpeg -ss 00:41:40 -i part02.mp4 -frames:v 1 -q:v 2 scratch\bilibili_covers_v1\part02_candidates\part02_2500.png
ffmpeg -ss 01:15:00 -i part02.mp4 -frames:v 1 -q:v 2 scratch\bilibili_covers_v1\part02_candidates\part02_4500.png
```

这里的时间点和文件名对应关系只是还原思路，不代表目录里保留了原始命令。

如果脚本里用秒数参数，也可能是：

```powershell
ffmpeg -ss 500 -i part02.mp4 -frames:v 1 scratch\bilibili_covers_v1\part02_candidates\part02_0500.png
ffmpeg -ss 1200 -i part02.mp4 -frames:v 1 scratch\bilibili_covers_v1\part02_candidates\part02_1200.png
ffmpeg -ss 2500 -i part02.mp4 -frames:v 1 scratch\bilibili_covers_v1\part02_candidates\part02_2500.png
ffmpeg -ss 4500 -i part02.mp4 -frames:v 1 scratch\bilibili_covers_v1\part02_candidates\part02_4500.png
```

### 4.3 候选帧 contact sheet

`part02_candidates/contact_sheet.png` 是 `960x620`，很可能是把 4 张候选缩小后排在一起。

这种 review 图非常有用，因为封面质量不是只看单张。

候选图横向比较时，要看：

- 哪张画面更亮。
- 哪张人物/场景更有代表性。
- 哪张左侧空白区更适合放字。
- 哪张色彩和其它 part 的封面能形成统一系列。
- 哪张缩略后不糊、不乱、不压字。

`part02` 最终选了 `source_frames/part02.png`，它和 `part02_candidates/part02_0500.png` 文件大小完全一致，尺寸也一致，说明 `source_frames/part02.png` 很可能就是从候选帧里复制或重命名来的。

## 5. 画布阶段

### 5.1 为什么从 1920x1080 扩到 1920x1200

原视频帧是标准横屏 `1920x1080`。

最终封面是 `1920x1200`。

这多出来的 120 像素不是随意加的。

它解决了三个问题：

1. 顶部可以加标题气氛区，不必把大标题压在原图最上边。
2. 底部可以加风险声明和倾斜色块，不必完全遮住原字幕。
3. 上传平台裁切时，有更大的安全边距。

如果直接在 `1920x1080` 上压字，容易出现：

- 主标题太靠上，被平台 UI 裁掉。
- 底部声明和视频字幕打架。
- 画面太挤。
- 标题卡片挡住太多原画。

`1920x1200` 给封面增加了一个“设计层”，而不是只在截图上硬贴字。

### 5.2 画布背景不是纯截图

成品封面除了源帧，还包含：

- 顶部黑色区域。
- 底部深色斜切区域。
- 右上角半透明主题色三角形或斜切块。
- 左上角主标题黑色圆角面板。
- 底部风险说明黑色圆角面板。
- 左侧主题色竖线。

这些元素共同把“游戏截图”变成“封面设计”。

如果只贴标题，没有这些遮罩，文字会和游戏画面互相干扰。

## 6. 排版结构

这批封面有统一结构。

可以拆成 7 个视觉层：

1. 背景截图层。
2. 顶部暗色遮罩层。
3. 右上角主题色斜切层。
4. 左上主标题信息卡层。
5. 右上分类胶囊层。
6. 中下部标签层。
7. 底部 beta 风险声明层。

### 6.1 背景截图层

背景使用 `source_frames/partXX.png`。

背景本身没有被 AI 重绘。

对于封面来说，背景截图承担的是“真实性”和“内容识别”：

- 用户看到这是真实游戏画面。
- 用户知道这不是泛泛的 AI 图。
- 用户能从截图判断视频内容是 Ghost Yotei。
- 字幕和水印也进一步暗示这是视频流程产物。

### 6.2 顶部暗色遮罩层

成品封面顶部有一块明显的黑色或深色区域。

它的作用不是装饰，而是提高标题可读性。

尤其是 `part02`，源图上半部分是蓝天。如果直接放白色标题，虽然也能看清，但缺少重量，标题不够“封面化”。加黑色区域后：

- `Ghost Yotei` 的白色标题更稳。
- 左上视觉重心更强。
- 和右侧亮色场景形成对比。
- 缩略图里标题不会飘。

### 6.3 右上主题色斜切层

每个 part 有自己的主题色：

- part01: 黄色 / 金色。
- part02: 绿色。
- part03: 红色。
- part04: 青蓝色。
- workflow beta: 黄色 / 金色。

这些颜色不是随便选的。

它们有几个作用：

1. 给分集建立视觉识别。
2. 避免五张封面看起来完全一样。
3. 让 `PART 01/02/03/04` 标签有明确记忆点。
4. 让右上角 `中英字幕` 胶囊和左侧竖线形成呼应。

斜切色块让画面不只是横平竖直的卡片堆叠。它给封面一点动势，但没有喧宾夺主。

### 6.4 左上主标题信息卡

左上主卡是封面最重要的元素。

典型内容：

```text
Ghost Yotei
PART 02
Codex 工作流 AI 试验 Beta
翻译校对 + 时间轴匹配 | 英语学习自用
```

它采用黑色圆角矩形作为底，文字以白色和主题色为主。

为什么这样排？

1. 大标题必须最大。
2. part 编号要贴近大标题，形成系列感。
3. `Codex 工作流 AI 试验 Beta` 是第二层信息，说明这不是正式商业字幕，而是流程实验。
4. 最后一行是用途和范围，字更小，避免抢主标题。

视觉层级大致是：

```text
Ghost Yotei                         最大
PART 02                             中等，主题色
Codex 工作流 AI 试验 Beta             中大
翻译校对 + 时间轴匹配 | 英语学习自用    中小
```

这就是为什么字体位置看起来比较舒服：它不是同一字号硬堆，而是分层。

### 6.5 左侧主题色竖线

主标题卡左边有一条主题色竖线。

它的作用有三个：

1. 把左上标题区和背景分隔开。
2. 呼应 `PART XX` 的颜色。
3. 让整个卡片看起来不是普通黑框，而像一个设计组件。

这个竖线是非常小但有效的元素。

没有它，卡片会显得笨重。

有了它，卡片更像视频节目包装。

### 6.6 右上 `中英字幕` 胶囊

右上角的 `中英字幕` 是一个圆角胶囊。

它承担的是“第一眼卖点”。

Bilibili 用户扫封面时，不一定先读副标题，但很容易看到右上角醒目的标签。

这个胶囊设计有几个要点：

1. 背景用 part 主题色。
2. 字体用黑色粗体。
3. 圆角较大，看起来像平台标签。
4. 放在右上，和左上大标题错开。
5. 它不遮挡左上主标题，也不挡住右侧主体太多。

### 6.7 中下部三枚标签

典型标签：

```text
英语学习自用
官方英文硬字幕
官方中文字幕
```

这些标签用小号黑底、白字、主题色描边。

作用是快速说明视频定位：

- 不是商业字幕发布。
- 有英文硬字幕来源。
- 有中文字幕来源。
- 适合自学 / 对照。

为什么不是放成一段长句？

因为封面缩略图里，短标签比长句更容易扫读。

标签的宽度根据文字长度自动变化，看起来就像 UI chip。这样比固定宽度按钮更自然。

### 6.8 底部风险声明

底部黑色面板写：

```text
非完整精校 | 字幕时间轴可能存在误差
workflow proof / beta release
```

这是这批封面很重要的一点。

它没有把项目包装成“完美官方字幕”。

它明确告诉观众：

- 这是 beta。
- 不是完整精校。
- 时间轴可能有误差。
- 它更像 workflow proof。

这和项目之前的 OCR / 对齐复盘是一致的：不要把 coverage 或自动化产物包装成完全正确的交付。

从封面设计角度看，这块声明也有好处：

- 填补底部空白。
- 平衡左上主标题重量。
- 给封面一个可信、克制的说明。

## 7. 字体选择

这批封面的字体看起来像使用了 Windows 本地字体，而不是生成式模型直接画出来的文字。

最可能的组合是：

- 英文标题：Arial / Arial Bold / Segoe UI Bold / DejaVu Sans Bold 这一类无衬线粗体。
- 中文标题和标签：Microsoft YaHei / Microsoft YaHei Bold / SimHei 这一类中文黑体。
- 小字说明：Microsoft YaHei Regular 或 Segoe UI。

为什么字体效果还行？

因为它避开了最容易出问题的做法：

1. 没有用花体字。
2. 没有用描边很重的综艺字体。
3. 没有把中文和英文强行塞进同一种比例很怪的字体。
4. 没有在复杂背景上裸放文字。
5. 大标题和说明文字字号差距足够大。

封面里 `Ghost Yotei` 的大标题非常粗，颜色接近暖白，不是纯刺眼白。中文主标题也是粗体，但字号明显小于英文标题。这样既照顾英文游戏名，也照顾中文说明。

### 7.1 为什么文字没有 AI 图常见错误

生成式图片模型直接画字时，经常出现：

- 多一笔少一笔。
- 英文拼错。
- 中文结构错误。
- 字间距失控。
- 局部糊掉。
- 小字不可读。

这批封面没有这些问题，是因为文字不是模型“画出来”的。

文字是用字体文件渲染出来的。

程序化文字渲染有确定性：

- 输入是什么字，输出就是什么字。
- 字号可控。
- 字重可控。
- 坐标可控。
- 行距可控。
- 文本不会凭空变形。

所以看起来反而比很多 AI 直接生成封面更可靠。

## 8. 颜色系统

### 8.1 基础色

基础色大致是：

- 背景黑：`#000000` 或接近黑。
- 面板黑：`#000000`，带透明或实色圆角。
- 主标题白：暖白，接近 `#FFF7E5`。
- 副标题白：纯白或浅灰。
- 小字灰：接近 `#D0D0D0`。

黑底 + 白字是封面可读性的基础。

### 8.2 分集主题色

不同 part 使用不同主题色：

- part01: 金黄。
- part02: 绿色。
- part03: 红色。
- part04: 蓝青。

主题色出现在：

- 左侧竖线。
- `PART XX`。
- 右上角 `中英字幕` 胶囊。
- 标签描边。
- 右上斜切色块。
- 底部斜切暗色区域。

这种重复使用让封面形成统一视觉系统。

### 8.3 为什么颜色不显得廉价

颜色没有大面积铺满整张图。

主题色只作为强调色使用。

主要大面积仍然是：

- 游戏截图。
- 黑色遮罩。
- 深色几何块。

这让封面有“游戏视频”质感，而不是变成纯 UI 海报。

## 10. 为什么没有直接用原生多模态能力输出图片

这里要区分两种“多模态”。

### 10.1 多模态理解

多模态理解是：

- 看图。
- 判断构图。
- 判断字幕位置。
- 判断文字可读性。
- 判断封面是否像缩略图。

这类能力适合用来做 review。

### 10.2 多模态生成

多模态生成是：

- 直接让模型生成一张封面图。
- 或者让模型修改图像。

这类能力在封面文字场景里有明显风险：

1. 中文和英文小字容易错。
2. 字体不稳定。
3. 版式不容易精确复现。
4. 需要改一个字时成本高。
5. 很难保证五张封面风格完全统一。
6. 很难保证不会篡改游戏截图细节。

所以这批封面没有走“直接生成图片”的路。

它走的是更工程化的路线：

> 多模态用于理解和判断，图像绘制用于确定性输出。

这也是为什么结果反而更稳。

## 11. OCR 和封面生成的正确关系

OCR 在这个项目里很重要，但它不应该承担封面排版任务。

更合理的分工是：

| 能力 | 适合做什么 | 不适合做什么 |
| --- | --- | --- |
| OCR | 读视频里的硬字幕 | 设计封面版式 |
| LLM | 总结文案、解释风险、生成标题候选 | 精确渲染文字 |
| 多模态理解 | 看截图质量、检查遮挡、评估可读性 | 批量稳定生成文字图 |
| Pillow / Canvas / Skia | 精确绘制文字和图形 | 判断哪张截图更有传播力 |
| ffmpeg | 抽帧、裁切、转码 | 判断封面美观度 |

这批图之所以“做得还行”，正是因为能力分工没有错位。

## 14. 自动换行与文本测量

这批封面的字位置之所以看起来比较稳，核心是文字绘制时应该做了文本测量。

程序化排版不能只写死每个矩形宽度。

正确做法是：

1. 用 `draw.textbbox()` 测量文字实际宽高。
2. 根据文字宽度计算 chip 宽度。
3. 根据字体高度计算垂直居中。
4. 如果文字超过最大宽度，则缩小字号或拆行。
5. 不同语言用不同字体 fallback。

例如标签 chip：

```python
def draw_chip(draw, x, y, text, font, accent):
    padding_x = 22
    padding_y = 10
    bbox = draw.textbbox((0, 0), text, font=font)
    text_w = bbox[2] - bbox[0]
    text_h = bbox[3] - bbox[1]
    w = text_w + padding_x * 2
    h = text_h + padding_y * 2
    draw.rounded_rectangle(
        (x, y, x + w, y + h),
        radius=18,
        fill=(0, 0, 0),
        outline=accent,
        width=3,
    )
    draw.text(
        (x + padding_x, y + padding_y - bbox[1]),
        text,
        font=font,
        fill=(255, 255, 255),
    )
    return x + w
```

这比手工估计宽度稳定得多。

## 15. 为什么标题位置看起来舒服

封面标题舒服，主要来自这些设计选择：

### 15.1 标题没有完全贴边

左边大约留了 100 像素。

顶部大约留了 90 到 120 像素。

这个边距让标题“站得住”。

如果标题贴到画布边缘，会显得廉价。

### 15.2 主标题卡有足够宽度

`Ghost Yotei` 很长，如果卡片太窄，会压迫文字。

现在卡片宽度大约超过 1000 像素，标题能完整展开。

### 15.3 字号层级很明确

`Ghost Yotei` 远大于其它字。

这让用户第一眼不会迷路。

### 15.4 信息块没有散

所有主要说明都归在左上卡片里。

右上只放一个短标签。

中下只放三个 chip。

底部只放 beta 风险。

每个区域只承担一个任务，所以画面不乱。

## 16. 为什么底部声明没有毁掉画面

底部声明一般很容易做丑。

这批图能接受，是因为：

1. 它没有放在画面正中央。
2. 它用了黑色半透明或实色面板。
3. 它的宽度有限，不横跨整张图。
4. 它和上面的标签有垂直间距。
5. 它用两行层级，中文风险大，英文解释小。
6. 它贴近左侧信息系统，而不是随机浮在画面上。

从传播角度说，底部声明牺牲了一点“纯净感”，但换来了诚实和项目可信度。

对于这个项目，这是值得的。

## 17. 和 Bilibili 封面环境的关系

Bilibili 封面不是海报。

它会出现在：

- 首页信息流。
- 搜索结果。
- 播放页推荐。
- 动态卡片。
- 移动端小尺寸缩略图。

所以封面必须满足：

1. 缩小后还能读出主标题。
2. 不依赖复杂小字。
3. 右上角/底部不能塞太多关键信息。
4. 明暗对比要足够。
5. 画面主体不能被标题完全盖住。

这批封面把最大标题放在左上，右侧保留游戏画面主体，是比较适合信息流的选择。

## 18. 当前封面的不足

虽然这批封面整体可以用，但还有改进空间。

### 18.1 底部原视频字幕和新增声明有时会竞争

成品里保留了原视频字幕。

这有语境价值，但也会带来干扰：

- 原字幕在底部中间。
- 风险声明在底部左侧。
- 水印在底部右侧。

三者同时存在时，底部信息较多。

如果以后做正式封面，可以考虑：

1. 抽没有字幕的帧。
2. 裁掉或模糊原字幕区域。
3. 用底部深色斜块统一压暗字幕。
4. 保留字幕但降低其视觉优先级。

### 18.2 `1920x1200` 上传后可能被裁切

如果平台最终强制裁成 `16:9`，上下 120 像素可能被裁掉。

所以重要文字要放在安全区。

建议安全区：

```text

y: 80 - 1080
```

如果底部声明放得太低，可能被裁。

### 18.3 小标签在移动端可能不够清楚

三枚小标签在大图里清楚。

但在移动端信息流里，它们可能只能作为视觉纹理，不一定能读清。

这不是致命问题，因为主信息仍然靠：

- `Ghost Yotei`
- `PART XX`
- `中英字幕`

小标签只是加分项。

### 18.4 水印仍然存在

底图右下角有 `MKIceAndFire` 水印。

这说明帧来自带水印的视频源。

如果封面用于公开发布，需要确认素材授权和平台规则。

从技术角度可以裁掉或遮住水印，但从合规角度不能简单把水印抹掉再假装没有来源。

## 22. 这批图做得好的地方

### 22.1 没有让 AI 直接画字

这是最关键的正确选择。

封面有很多中文字和英文字，如果直接用生成式图片模型，文字很容易坏。

程序化绘制保证了字是对的。

### 22.2 保留了真实游戏画面

真实帧比 AI 生成背景更可信。

用户能看到实际视频内容，不会误以为是一个泛泛的宣传图。

### 22.3 视觉系统统一

五张封面结构一致：

- 左上大标题。
- 右上字幕标签。
- 中下功能标签。
- 底部 beta 声明。
- part 主题色。

这让它们像一个系列。

### 22.4 对风险有说明

封面没有假装“完美字幕”。

这和项目交付监督思路一致。

### 22.5 字体和留白比较克制

没有过度装饰。

没有一堆花哨描边。

没有把所有字都做成最大。

这让它看起来更像工程项目封面，而不是廉价营销图。

## 23. 这批图为什么比“纯自动封面”更好

很多自动封面会失败在：

- 抽帧随机。
- 字太小。
- 背景太乱。
- 标题和人物打架。
- 没有系列统一性。
- 每张图风格漂移。

这批图避免了这些问题，因为它不是纯自动。

它是“半自动设计”：

- 抽帧可能半自动。
- 候选需要看 contact sheet。
- 模板是程序化固定的。
- 文案是人为/LLM 辅助确定的。
- 最后通过视觉 review。

这也是我更推荐的路线。

对于这类字幕工作流项目，最可靠的方法不是让模型直接产出最终图，而是让模型帮助做决策，然后用确定性程序输出。

- 避开过暗场景。
- 优先有角色、地标、动作、情绪的画面。
- 保留字幕版和无字幕版各一套，如果源视频允许。

### 24.2 第二阶段：候选评分

用脚本做基础评分：

- 亮度均值。
- 对比度。
- 边缘密度。
- 黑边比例。
- 人物/主体大致区域。
- 左上区域复杂度。
- 右上区域复杂度。

这一步不需要 OCR。

### 24.3 第三阶段：人工或多模态 review

让多模态模型或人工看 contact sheet：

- 哪张最有点击欲。
- 哪张最适合放标题。
- 哪张缩略图最清楚。
- 哪张最能代表该 part。

多模态能力适合在这里用。

### 24.4 第四阶段：模板渲染

用固定模板渲染：

- 统一字号。
- 统一色板。
- 统一安全区。
- 统一标签。
- 统一风险声明。

### 24.5 第五阶段：缩略图回归测试

生成：

- 原图。
- `480x300`
- `320x200`
- `240x150`
- contact sheet。

然后一起检查。
